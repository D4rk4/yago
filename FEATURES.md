| FTR-007 | Storage | Embedded durable storage and quota pressure | Implemented with replaceable storage boundaries, document, RWI, and URL metadata collections, concurrent transaction safety, eviction, `yago-node.db` for new data dirs, and fallback opening of existing `yacy-rwi.db` files. The vault engine is sharded and compressed (ADR-0025): records route by xxhash64(bucket+key) across N quota-derived bbolt shard files (one per ~7 GB, min 8, power of two, layout pinned in vault/manifest.json) living in a three-level fanout (vault/aa/bb/cc/aabbcc.vlt), keeping every file far below the 10 GB cap; values compress with zstd at the fastest level (frame checksums verify on read; small/incompressible values stay raw with a crc32c) while keys stay uncompressed so ordered prefix scans work — reads merge the shards' cursors into one ascending key sequence. Each shard is an independent failure domain: losing one file loses 1/N of the keyspace, never the store. Cross-shard atomicity is relaxed by design (per-shard commits; a failed commit rolls back the remaining shards so none stays write-locked). A legacy single-file vault migrates automatically on startup (compressing on the way) and is retired to a .migrated.bak file. The full-text search index shards the same way (ADR-0025): documents route by xxhash64(docID) across 8 scorch shard indexes in the three-level fanout (index root/xx/zz/yy/xxzzyy.idx), each created with a merge policy capping merged segments at 400k documents (zap files stay far below the 1 GB cap; zap segments are already snappy-compressed and this mapping stores no bodies) and three retained snapshots; searches fan in through a bleve IndexAlias so callers still see one index, counts sum across shards, and a legacy single-directory bleve index rebuilds into the sharded layout from the vault on startup (or keeps serving as a single compatibility shard when no rebuild source exists).
