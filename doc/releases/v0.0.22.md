## Abstract

YaGo v0.0.22 corrects Tavily-compatible availability, web-result materialization, and gRPC transport security. Every depth now uses the bounded global local-plus-peer pipeline; verification remains depth-specific and operator DDGS policy remains authoritative. Zero-row partial-source failures are retryable only while the original caller remains live; inherited cancellation or deadline keeps the infrastructure-error path. Domain constraints narrow local and provider work without replacing final hostname checks. Web discoveries gain recoverable durable admission, active-only URL coalescing, and live-lease profile authorization, allowing successful crawler ingest to become document-backed local content and enter recrawl scheduling. Routes, settings, public schemas, listeners, and deployment topology are unchanged. Internal crawl-broker state gains additive keyspaces and optional lease fields; no rebuild is required, but node and crawler should be upgraded together.

## Changes

### Tavily-compatible global retrieval and availability

- `basic`, `fast`, `ultra-fast`, and `advanced` now use the same bounded global local-plus-peer retrieval pipeline. The first three depths retain `verify=false`; `advanced` retains `verify=ifexist`.
- The existing web-fallback policy remains authoritative. `always` starts DDGS beside local and peer retrieval for every supported depth; miss-triggered modes still wait for the applicable local-plus-peer and bounded local-recovery miss.
- A search that returns no rows while one or more sources remain incomplete now answers HTTP 503 with `Retry-After: 1` and the existing detail-only Tavily error envelope while the original caller context remains live. A child or source deadline is therefore retryable while that caller remains live; inherited caller cancellation or deadline retains the existing infrastructure-error path. An honest complete miss remains HTTP 200 with an empty `results` list, and completed rows with non-fatal partial-source evidence remain a successful response. Raw-content search maps the same availability state without changing its response schema.
- Non-facet local retrieval omits the relaxed candidate pass only when the strict pass fills the requested window and reports more strict rows than that window. The exact pagination boundary retains relaxed retrieval. Pseudo-relevance feedback omits its second pass when the smaller of the request limit and the fixed 50-row activation ceiling is already full.

### Domain-constrained local and web retrieval

- Tavily `include_domains` may narrow the outbound DDGS request with `site:` operators for DNS names and IPv4 literals when the complete URL-encoded provider query fits the fixed 2,048-byte ceiling. IPv6 literals and larger expressions retain the bounded base query rather than sending an ambiguous, truncated, or syntactically partial constraint.
- Include domains also add a safe Bleve URL-field candidate prefilter. A value without a usable URL analyzer token bypasses this optimization so recall remains under the existing stored-host filter.
- Local, peer, and web results still pass the authoritative normalized hostname-suffix checks before source fusion and response emission. DNS names, IPv4 literals, and bracketed IPv6 literals use canonical host forms; exclusion continues to win over inclusion.
- Provider cache identity now includes verification mode and every structural constraint, preventing a row set accepted for one request policy from being reused under another. `verify=false` suppresses the additional visible query-term test but never bypasses URL, domain, site, TLD, file-type, in-URL, or exclusion constraints.

### Recoverable web-result materialization

- Optional web warming still runs outside the search response through the bounded two-worker, 128-pending-job queue. An eligible normalized URL now persists a node-internal recovery intent before automatic-discovery queue mutation. Startup and the next admission reconcile an unfinished intent idempotently.
- Separate active, pending, and leased indexes coalesce only live automatic-discovery work. The ordinary new-URL and indexed leased-URL paths do not enumerate the complete queue. Legacy recovery may perform a compatibility scan when its indexes are missing.
- Acknowledgement, terminal completion, and cancellation first persist a typed settlement intent, then apply the destructive lease transition, converge every settlement index and the active URL identity, and finally remove the intent. After that durable stage, a one-second internal recovery budget no longer inherits RPC cancellation; an expired internal budget retains the intent for retry or startup. Plain negative acknowledgement, expiry requeue, and worker-session adoption resolve a staged intent before they can mutate its lease. A conflicting disposition cannot replace the first durable intent.
- Requeue retains the normalized identity. Settlement history reserves an unoccupied sequence before publication and relocates an older partial record if an interleaved settlement claimed that sequence, repairing its cursor, order, and expiry indexes idempotently. A later search can therefore retry completed or failed work without creating a second root fetch while work is active. Manual permanent idempotency keys remain in their existing namespace.
- Recovered admission and duplicate-lease removal signal the order stream after durable queue progress, so an available pending order cannot remain asleep behind recovery or coalescing.
- Ingest authorization now binds lease ID, worker session, run provenance, and profile handle to the live leased order and supplies that order's complete profile to the ingest consumer. A stale or unrelated session cannot select a profile by handle alone.
- Dispatch attempts to persist the exact format-stamped profile before publishing the order, retains at most 256 successful handles in FIFO order, and retries a failed or evicted registration on a later dispatch. An evicted handle is re-registered through another idempotent profile write, and a successful changed same-handle write precedes publication of that later order. Successful ingest attempts to fill a missing profile before recording the fetch; an older in-flight lease cannot overwrite a newer stored profile. A grouped ingest attempts to schedule every URL from its own live-lease cadence even when several leases share one handle with different intervals. Profile and schedule writes remain best-effort after document, metadata, and posting persistence: failures are logged and cannot roll back an indexed page.
- Crawl-format settings are loaded once when their store opens. Reads use the in-memory snapshot, while an update changes the snapshot only after its durable write succeeds.

### Transport security maintenance

- The node, crawler, crawl contract, and both Docker E2E modules now pin grpc-go v1.82.1, the first release patched for `GHSA-hrxh-6v49-42gf`. Its xDS RBAC authorization and panic corrections plus HTTP/2 control-frame throttling replace the affected v1.79.3 transport without changing YaGo RPC schemas or service topology.

## Compatibility

- No environment variable, Admin setting, route, request field, response field, listener, port, service, image name, package path, Bleve mapping, document encoding, or deployment topology changes.
- The existing grpc-go dependency advances from v1.79.3 to the first security-patched v1.82.1 release. Generated RPC schemas and the node-crawler wire contract are unchanged.
- The observable Tavily scope changes: `basic`, `fast`, and `ultra-fast` can now use peers and the operator-enabled DDGS policy instead of remaining local-only. Existing request and success envelopes are unchanged. A previously ambiguous zero-row result can now be HTTP 503 when source failures prove that it was not an honest complete miss and the original caller remains live; inherited caller cancellation or deadline retains the infrastructure-error path.
- `verify=false` continues to accept provider relevance without requiring every query term in the returned title, snippet, or URL. Structured request constraints remain mandatory. `advanced` retains bounded visible query evidence through `verify=ifexist`.
- The public portal and equivalent Tavily `advanced` requests retain their canonical ordering contract when click-exposure randomization is disabled. Other depths share the retrieval scope but retain their different verification policy.
- The crawler RPC and JSON contracts are unchanged. The node derives the authoritative profile from the existing leased order rather than adding a crawler-supplied field.
- Automatic-discovery admission intents, settlement intents, and active, pending, and leased indexes are additive internal keyspaces. Lease JSON gains optional internal fields; records written by v0.0.21 omit them safely. Existing durable orders, leases, frontier records, crawl profiles, documents, URL metadata, and RWI postings remain readable.

## Upgrade and rollback

1. After the immutable tag workflow succeeds, download the architecture-matching v0.0.22 Debian package. Verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.22`, and the tagged source commit before stopping either service.
2. Follow the target's data-protection policy. Generic bare-metal nodes take one coordinated stopped backup. The documented `root@yagoseek.dev` exception preserves every existing archive under `/opt/yago/backups` and creates no new pre-upgrade backup; stage the verified package under `/opt/yago/releases`.
3. Stop `yago-crawler` and then `yago-node`. Install the package with `dpkg -i` and confirm that package installation did not start either service automatically.
4. Start `yago-node` first. Verify the exact version, `/health`, `/ready`, configured listeners, current logs, and recovery of the durable crawl queue.
5. Start `yago-crawler` after node readiness. Verify its exact version, control and ingest connections, a current heartbeat, progress on a newly admitted automatic-discovery URL, and absence of repeating lease-loss output.
6. Keep production verification observation-only. Exercise source failure, web materialization, queue recovery, and independent process restart scenarios in the current-source local Docker gate rather than against the production node.

For rollback, stop crawler and node, reinstall the v0.0.21 package, start and verify the node, and then start the crawler. No index rebuild or data restore is required solely for format compatibility. The older node ignores the additive discovery keyspaces and unknown optional lease fields. It can therefore lose active-only coalescing evidence and admit a temporary duplicate automatic-discovery order while running; existing orders remain processable. A later v0.0.22 start retains the additive indexes it understands, while rediscovery repairs pending or leased automatic work whose optional ownership fields v0.0.21 rewrote away. Unfinished v0.0.22 admission or settlement intents remain ignored during rollback and are reconciled if v0.0.22 is started again. The older release does not restore v0.0.22 global-depth availability, request-aware provider filtering, or lease-authorized profile safeguards.

## Known limitations

- A web result is discovery evidence, not a stored document. Its first response can carry `raw_content: null`; it becomes document-backed only after optional warming is enabled, the bounded warming queue accepts it, a crawler is connected, and the normal egress, robots, content, quality, storage, and index gates succeed.
- A full 128-job warming queue warns and drops only the new optional warming job. External provider failure, rate limiting, an unavailable crawler, or a permanently rejected target can delay or prevent local materialization; a later surfaced URL can retry after active work settles.
- `basic`, `fast`, and `ultra-fast` intentionally use `verify=false`. They retain provider-selected lexical relevance and can return a structurally eligible row whose visible title, snippet, and URL do not contain the literal query wording.
- Provider `site:` pushdown is omitted for IPv6 literals and when the complete URL-encoded query would exceed 2,048 bytes. Final hostname filtering remains correct, but a provider can return fewer permitted rows than the requested limit after disallowed rows are removed.
- The Bleve include-domain candidate query is an optimization, not the authority. Values without a usable URL analyzer token retain the complete bounded local candidate path.
- Profile and recrawl-schedule writes are best-effort after successful ingest. A write failure leaves the document local and searchable but can leave that fetch without a durable profile or future recrawl entry until a later dispatch or ingest supplies one.
- Every Tavily depth now permits bounded peer and operator-enabled web work, so the local-only latency profile of v0.0.21 basic, fast, and ultra-fast searches no longer applies. Existing interactive deadlines and process-wide provider limits remain unchanged.
- Rollback to v0.0.21 preserves readable queue data but does not understand the additive active-only coalescing indexes; duplicate automatic-discovery work can be admitted while rollback is active or until later rediscovery repairs work rewritten by the older node.
