## Abstract

YaGo v0.0.18 makes planned restarts observable and bounded, raises crawl-ingest aggregation while limiting Bleve write concurrency, and closes operational gaps found after v0.0.17. Clean shutdowns checkpoint bbolt freelists; crawler state gains live 4 GiB soft boundaries and startup copy compaction; index rebuilds gain storage preflight and journal progress; fleet permits account for bounded control-plane latency; browser-pool metrics and first-sample Admin gauges become immediately useful; and operators can queue extraction-generation refreshes in bounded batches. Tavily `include_usage` now reports request-local compatible units. Two new environment bootstraps have matching Admin controls; the frontier value is synchronized across node and crawler configuration. No port, listener, service, YaCy wire, or automatic-recrawl change is introduced. Rollback still requires a coordinated stopped backup.

## Changes

### Main-vault restart lifecycle

- The serving path continues to use bbolt with freelist persistence disabled, avoiding per-write freelist serialization. A clean shutdown now writes one fully synced empty transaction per shard so the in-memory freelist and accepted writes are durable before that shard closes.
- A failed checkpoint does not stop the remaining shards from closing. Deferred-fsync mode additionally attempts a best-effort shard sync, and all checkpoint, sync, and close failures are returned together.
- The first start after upgrading from an older release still reconstructs freelists by scanning. A start after a successful v0.0.18 clean shutdown can load the persisted freelists; an unclean stop or failed checkpoint deliberately returns to scan-based recovery.
- Startup emits stable structured records on standard output before and after every sequential shard open. Records carry the human shard ordinal, completed and total shards, path, byte size, and elapsed time. RWI word-filter construction is a separate phase; successful completion is INFO and a degraded completion is WARN with the affected-shard total.
- Event persistence receives five seconds to drain and five seconds to leave its transaction after cancellation. If it still remains active, shutdown reports an error and skips vault close rather than closing over the writer or leaving a close goroutine behind.
- The shipped systemd unit and Compose example now allow up to 15 minutes for an otherwise orderly node stop so large vaults can finish bounded writer quiescence and shard checkpointing.

### Crawl-ingest persistence

- The node can coalesce up to 64 ready crawl deliveries and 64 MiB of their encoded JSON, compared with 16 deliveries in v0.0.17. A partial group waits at most ten milliseconds and stops waiting on cancellation.
- One encoded batch that exceeds the contract ceiling is rejected before JSON decoding. Per-delivery admission, acknowledgement, arrival order, deterministic duplicate coalescing, retry, and backpressure semantics remain intact.
- Posting intake from one group is split into retryable chunks of at most 8,192 rows, so a full group cannot become one 524,288-row storage transaction.
- Populated Bleve shards persist through at most four concurrent commit lanes rather than an eight-shard write burst. Every touched shard finishes before the deterministic lowest-shard error is returned.
- An empty index batch is a strict no-op. Every nonempty attempt invalidates completed search cache entries even when it fails, because an earlier fallback document or shard may already be durable.

### Crawler-state boundaries and startup compaction

- `crawler.node_state_max_bytes`, bootstrapped by `YAGO_CRAWLER_NODE_STATE_MAX_BYTES`, adds a live 4 GiB default soft physical boundary for the node's `crawlbroker.db`. Zero disables the boundary.
- At or above the node boundary, fresh order publication is rejected. Duplicate detection, migration, ingest, recovery, lifecycle, settlement, and cleanup writes remain available.
- `crawler.frontier_state_max_bytes`, bootstrapped by `YAGO_CRAWLER_FRONTIER_STATE_MAX_BYTES` in both service environments, adds the corresponding live 4 GiB default boundary for `crawler/frontier-v1.db`. Zero disables it.
- At or above the frontier boundary, a fresh order waits before seed expansion and wakes after the value is raised or disabled. An already committed seed manifest completes, existing pages continue, and recovery and terminal work remain available. New discovered-link batches are refused instead of being retained for later admission.
- When an enabled state file starts at or above its boundary, the owning process takes a persistent path-stable sidecar lease, remeasures the source, reserves that temporary headroom through the shared maintenance gate, and writes a private sibling bbolt copy. The copy is synced and atomically installed before the containing directory is synced and the authoritative database is opened.
- A failure before replacement leaves the original file authoritative and startup continues with a warning. A directory-sync or final-inspection failure after replacement reports that the compacted copy is installed with a durability warning. Failure to acquire or release the sidecar lease stops startup.
- The node sends the frontier boundary and crawler storage reserve/hysteresis as additive, presence-aware startup policy fields, so a current crawler applies persisted Admin policy before checkpoint maintenance. An older node that omits a field leaves the crawler's bootstrap or current value intact.

### Fleet fetch-start delivery

- The existing fleet-wide `crawler.max_pages_per_second` ceiling remains non-bursting and node-authoritative. A crawler now measures one completed permit RPC from request start through response receipt, caps that observation at one second, and reports it as the delivery allowance for the next sequence.
- The node widens the next demand-backed permit batch by that prior allowance and reserves its complete final window before scheduling another batch. The crawler intersects each opening with response receipt and each closing with request send, then enforces the configured interval between permits it actually consumes.
- A latency spike larger than the preceding allowance may discard a window but cannot produce a catch-up burst. Its measurement applies to the following sequence. Omitted or zero allowance retains the conservative v0.0.17 behavior.

### Extraction-generation refresh

- Every newly parsed live document carries extraction generation `1`; zero remains the omitted legacy value. The stored document and Admin detail view retain and display both stored and current generations.
- Admin → Index adds an explicit action that examines 1 through 100 raw storage records per request across the legacy and admission-ordered partitions. It queues only visible documents below the current generation through the existing durable crawler dispatcher and reports examined, visible, outdated, current-or-newer, newly queued, and already accepted totals.
- One pass captures the initial high key of each partition, so continuous ingest cannot extend it indefinitely. The bounded continuation carries the action identity, scan position, limits, and dispatch identity. A read failure or indeterminate dispatch keeps the input position for an idempotent retry.
- Extraction refresh is never started automatically and has no environment or runtime setting. A material parser change advances the shared generation constant; an operator then runs the bounded action when recrawl timing and capacity are acceptable.

### Index rebuild and operator telemetry

- Before a scheduled or mapping-driven Bleve rebuild deletes the old index, startup counts stored documents, measures the current index footprint when available, and requests that measured footprint as additional filesystem headroom. Rejected admission preserves the old index and durable rebuild marker and stops startup with an actionable error.
- An admitted rebuild logs one INFO start record, at most the nine 10-percent milestones from 10 through 90 percent, and one completion record with indexed documents, batches, and elapsed time. Rebuild batches remain bounded at 16 documents.
- The Admin metric sampler now takes one synchronous gathering after queue collectors are registered. Queue, memory, and storage gauges can render as current-value cards immediately; sparklines and counter-derived request rate, error rate, latency, DHT rate, and process CPU still require the second successful sample, normally about ten seconds later.
- Crawler Prometheus metrics add `yacy_crawler_browser_slot_acquisition_seconds`, `yacy_crawler_browser_sessions{state="ready|busy|cooling"}`, and `yacy_crawler_browser_failures_total{reason="slot_deadline|cooldown|launch|render"}`. Every allowed state and reason starts at zero; unknown reasons are discarded, and URLs and raw error text never become labels.
- The existing `yacy_crawler_browser_slot_acquisition_deadlines_total` remains available and advances with the `slot_deadline` reason.

### Tavily-compatible usage

- When `include_usage` is true, Search, Extract, Crawl, and Map responses now include request-local compatible units derived from completed work.
- An executed basic, fast, or ultra-fast search reports one unit; advanced reports two. A `max_results: 0` request that does not execute search reports zero.
- Extract reports `floor(successful results / 5)` units and applies a twofold multiplier for advanced depth. Failed extraction rows do not count.
- Map reports `floor(successful pages / 10)` units and doubles that value when instructions are present. Crawl adds those mapping units to `floor(successful pages / 5)` extraction units, with the advanced extraction multiplier where selected.
- Usage remains endpoint-local compatibility accounting. It is not billing, an account balance, external-provider expenditure, or evidence that an upstream Tavily service was called.

## Compatibility

- Upgrade `yago-node` and `yago-crawler` together from the same v0.0.18 package. Finite fleet-rate operation deliberately fails closed when either side lacks the current fetch-start lease capability.
- The node-crawler contract changes are additive: permit delivery allowance uses a bounded optional field; runtime-policy fields 18 through 20 carry the frontier boundary and startup storage reserve/hysteresis; extraction generation is additive JSON inside the unchanged ingest protobuf envelope. Older protobuf or JSON decoders ignore unknown fields.
- Generation zero preserves legacy documents and payloads. No automatic document migration or recrawl occurs during upgrade.
- YaCy peer endpoints, public search shapes, existing Tavily fields, Admin routes, listener addresses, service topology, and ports are unchanged. Tavily usage appears only when requested.
- Two new environment bootstraps and matching live Admin settings are introduced: `YAGO_CRAWLER_NODE_STATE_MAX_BYTES` and `YAGO_CRAWLER_FRONTIER_STATE_MAX_BYTES`. Keep the frontier bootstrap equal in the node and crawler environments until typed policy becomes authoritative.
- The systemd and Compose stop grace changes are deployment-default changes, not new processes or listeners. No third-party runtime dependency is added.
- This release changes no Bleve mapping and does not schedule a full-index rebuild merely by being installed. The new preflight and progress behavior applies when an existing rebuild marker or a future mapping change requires rebuilding.
- State-file copy compaction preserves the current logical bbolt schema. The extraction-generation member is backward-readable as an ignored field, but coordinated rollback remains required because the node broker and crawler checkpoint must represent one recovery boundary and older releases can observe stale retained cutover state.

## Upgrade and rollback

1. Download the architecture-matching published v0.0.18 package and verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.18`, and the tagged source commit.
2. Take one coordinated backup of the complete node and crawler data while both services are stopped. Keep the archive outside the live data and release-staging directories.
3. Confirm sufficient filesystem headroom. A state file at or above its enabled boundary can require a temporary sibling copy approximately as large as the current source, in addition to the configured reserve.
4. Stop `yago-crawler`, then stop `yago-node`. Install the matched package containing both binaries with `dpkg -i`; package installation enables but does not start either service.
5. Start `yago-node` first. Watch standard output through the service journal for sequential shard-open, word-filter, optional crawl-state compaction, and any rebuild-preflight records. Verify `/health`, `/ready`, filesystem headroom, and the configured crawler-state boundaries.
6. Start `yago-crawler` only after node readiness. Verify its exact version, runtime-policy connection, advancing crawler heartbeat, frontier state, fleet-rate behavior, and browser-pool metrics when the loopback metrics listener is enabled.
7. Use Admin → Index extraction refresh only when the crawler has acceptable capacity. Continue bounded batches until the action reports completion; do not assume the upgrade queued old documents automatically.

For rollback, stop the crawler and node, restore the coordinated pre-upgrade backup, install both older binaries, start the node, verify readiness, and then start the crawler. Do not delete only `crawlbroker.db`, reuse the retained legacy cutover rows, or combine a restored node broker with a newer crawler checkpoint. The state-file compaction itself needs no format conversion, but an in-place binary downgrade does not provide a coherent crawl recovery boundary.

## Known limitations

- The two crawler-state values are soft physical admission boundaries, not filesystem quotas. Existing work, lifecycle and recovery writes, bbolt free pages, and work accepted just below a boundary can leave a file at or above it. Use a filesystem, project, or volume quota for a hard aggregate limit.
- Startup compaction needs temporary free space approximately equal to the remeasured source file. A pre-install copy failure leaves allocated free pages in the original file; a post-install directory-sync warning means the compacted file is installed but its rename durability was not fully confirmed.
- The first node start after upgrading still reconstructs bbolt freelists, and the RWI word filters rebuild on every start. A failed checkpoint, forced termination, or uncooperative event writer also returns the next start to scan-based recovery. The 15-minute stop allowance is a ceiling, not a shutdown-duration guarantee.
- Bleve rebuild headroom uses the current index footprint as an estimate. Segment creation and merging can require a higher transient peak. Rebuild progress is journal evidence, not an Admin progress control or resumable checkpoint; interruption restarts the rebuild from the beginning.
- Extraction refresh is manual, generation-based, and bounded. Its separate reads are not one transactional snapshot, so concurrent replacement or deletion may change what a pass observes; admissions after the captured high keys wait for a later pass. It performs network recrawls and remains subject to ordinary robots, egress, format, rate, and failure policy.
- The fleet permit allowance uses the previous completed RPC. One new latency spike can still discard the current window; the design prevents catch-up bursts rather than guaranteeing full use of every configured slot.
- Browser metrics expose four fixed failure classes, not raw browser errors or per-URL diagnosis. They require the optional loopback crawler metrics listener and a separate trusted proxy or tunnel for remote scraping.
- Tavily usage rounds down to complete five-result and ten-page groups. It is compatibility metadata only and does not implement billing, account balances, provider expenditure, model-guided crawling, semantic reranking, or an upstream Tavily provider.
- Cross-shard bbolt and Bleve effects remain non-atomic. Reservations, retry, cache invalidation, deterministic error selection, and bounded groups limit partial failure; they do not create one transaction across every shard, index, document, anchor, RWI, and URL-metadata store.
- Local validation used an approximately 220 MiB state set and a synthetic 65,536-document sustained comparison. It does not establish production-scale startup duration, compaction duration, free-space demand, or interactive-search latency on another storage device.
