## Abstract

YaGo v0.0.21 hardens YaCy peer interoperability, remote retrieval, and outbound DHT durability. Seed-hash identity excludes the local node without confusing peers behind the same address; bounded durable observations support alternate-address transport. Query, seed-list, hello, and search adapters more closely follow stock wire semantics. Remote resources require a matching WordReference before side effects, and partial redundant DHT handoffs remain recoverable after local or process failure. Mail and news storage enforce deterministic retention. Exact `site:` matching remains separate from Tavily suffix filtering. Routes, settings, dependencies, listeners, services, images, crawler contracts, primary peer-row format, and index layout are unchanged. Additive lifecycle metadata is ignored by v0.0.20. No migration or rebuild is required; upgrade both packaged processes together.

## Changes

### Seed-hash identity and durable peer lifecycle

- The immutable seed hash is now the sole self-identity boundary. Startup removes a legacy persisted self row, every roster ingress rejects that hash, and lookup, counting, candidate, active-membership, eviction, seed-list, search, DHT, and Admin projections all exclude it. Endpoint equality no longer defines self identity, so distinct peers behind one NAT remain eligible.
- A directly verified hello response refreshes the responder's current seed data only after the response hash identifies the contacted peer. The endpoint that actually answered becomes primary; unverified seed-list gossip remains first-write and cannot downgrade that direct observation.
- Authenticated hello callers receive their locally measured callback type. A successful callback retains or promotes a caller; a failed callback retains it as junior under a separate bounded persistence context. Junior peers remain visible in the durable roster and Admin peer list but do not enter reachable membership, outbound search or DHT routing, or advertised seed sets.
- Transport failure moves an observed peer into a ten-minute cooling interval instead of deleting it. The peer remains in the bounded 24-hour durable reservoir and can be promoted by a later successful observation.
- Exact normalized host-and-port ownership permits one routable claimant. Verified evidence takes precedence over unverified gossip, without using DNS aliases or a shared endpoint as peer identity.
- Outbound hello accepts a usable literal `yourip` or the DNS hostname advertised by the supplied local seed.
- Bootstrap refresh and request-time roster reads remain bounded. Bootstrap refresh runs all configured sources with at most eight concurrent fetches under one ten-second deadline, rejects future or older-than-24-hour observations, chooses the freshest duplicate hash, and retains at most 4,096 seeds and 16 MiB. Roster lookup, seed-list, and name resolution share an active-first bounded snapshot instead of scanning durable storage on each request. Legacy roster-capacity initialization uses one complete value scan with bounded retained memory, followed by key-only deletion pages.
- Public reachability remains an independently reported tri-state. Current peer evidence is authoritative, but inbound public reachability is not an outbound DHT sender gate, matching stock YaCy behavior. The self seed begins as virgin after process start, becomes junior after current failure-only evidence, and becomes senior after current successful evidence.

### Bounded alternate-address transport

- Outbound hello, inbound hello callback, remote search, and outbound RWI and URL transfer can use up to five unique advertised hosts in deterministic primary-`IP` then `IP6` order.
- All candidate addresses share the caller's original aggregate operation deadline. Each remaining address receives a bounded share of the remaining time; an unresponsive address cannot restart or multiply the operation timeout.
- Hello and callback operations can continue after transport failure or any unusable response. Remote search continues only after a `Client.Do` transport failure or response-body I/O failure; an HTTP status, size, parse, abstract, or WordReference failure stops fallback for that logical response.
- RWI and URL transfer continue only when `Client.Do` returns a transport error. Once an HTTP response is received, its status, body-read, size, or parse outcome ends address fallback for that transfer.
- IPv6-only seeds are eligible, and IPv6 URL authorities are bracketed. A verified successful address can become primary while numeric alternatives remain available. The implementation does not resolve DNS aliases as identity evidence.

### Authenticated requests, hello, query, and seed-list wire parity

- `/yacy/query.html` treats `iam` and `youare` as opaque form values, authenticates the exact caller value, and requires `youare` to identify the local seed. A missing `iam` remains acceptable on an open network.
- Per-word `rwiurlcount` uses the requested word hash, `lurlcount` is the stock constant `1`, and unknown objects or failed per-word index reads emit the stock unresolved-template marker instead of an invented numeric answer.
- Plain seed lists emit stock compact `b|` and `z|` rows, retain unique primary-first addresses, and preserve selector-presence semantics for `my`, `me`, and related filters. Active peers precede recently passive peers.
- Seed-list `maxcount` and hello `count` share Java-compatible signed-decimal int32 parsing. BMP decimal digits use `Character.digit(char, 10)` values, only ASCII signs are accepted, supplementary digits are invalid, and malformed or overflow values select each endpoint's stock fallback. Version floors use Java-compatible binary32 parsing. Stored seed versions use Java-compatible binary64 grammar; missing, malformed, and numeric-zero versions retain upstream developer-version eligibility.
- Extension-derived response types now match stock routes: query and plain seed-list HTML responses use `text/html`, JSONP seed lists use `application/json`, and unrelated YaCy wire routes retain their existing MIME values.
- Every authenticated YaCy request adapter preserves whether `network.unit.name` was absent or explicitly empty. A missing name defaults to `freeworld`, an explicitly empty value remains empty and is rejected against `freeworld`, and an empty local configuration still denotes `freeworld` in open and salted modes.
- Hello preserves `iam` as an opaque wire value and authenticates its exact bytes. Its raw seed is accepted through 16,000 Java `String.length()` UTF-16 units and rejected above that boundary before generic seed decoding. Authentication failures and self requests return the stock virgin/no-seed envelope. A successful response contains the local seed plus at most 100 reachable peers ordered by observed freshness with a stable hash tie-break; a nonpositive count returns the local seed without additional peers, and a reachable principal caller remains principal.

### Partition-safe remote search

- Outbound `/yacy/search.html` supplies matching `myseed` and `iam` values and transports directly representable language, site-host, author, file-type, URL-filter, and preference fields.
- Inbound RWI search applies bounded author containment, URL regular-expression, file-extension, and scheme constraints to the joined metadata candidates before top-result selection. URL expressions are limited to 2,048 bytes and use Go's linear-time RE2 engine.
- An authenticated search request may contribute one transport-address-only virgin candidate when `myseed` and `iam` identify the same non-self seed. This observation cannot claim reachability, overwrite a known peer, trust an advertised host, or add a network round; ordinary hello remains the promotion path.
- DHT target selection retains vertical-partition coverage before adding redundancy. With the default exponent `4` and primary limit `16`, all 16 partitions remain represented when eligible peers exist.
- One query starts at most 32 physical peer HTTP attempts. Logical resource calls, morphology probes, and physical address attempts have separate bounded accounting while retaining the existing 1.3-second remote-search boundary.
- Multiword primary calls use stock `abstracts=auto`. Exact secondary recovery consumes the returned abstracts rather than issuing a separate exact-term probe, and sends at most one stock-compatible secondary URL request to each peer.
- Resource, abstract, and secondary calls feed one request-scoped peer lifecycle reduction. A valid success wins over a transport failure for the same peer; invalid protocol data changes peer reputation without incorrectly changing reachability.

### Remote-resource integrity

- Every received search resource must carry an enhanced-base64, fixed-order 20-column YaCy `WordReferenceRow` in `wi`, and that row's URL hash must equal the resource URL hash.
- One malformed row rejects the peer response before resource counters, result conversion, ranking, optional document-cache warming, or crawler seeding. The event is recorded as invalid protocol evidence rather than successful traffic.
- Validated peer metadata may still enter the existing bounded optional document and full-text cache without overwriting a locally crawled document. The peer-attached RWI row is not copied into the local exchange vault because it has no durable sender provenance or expiry.
- Peer `references` remain untrusted hints. They are not injected into ranking or query expansion; locally derived response topics remain unchanged.

### Crash-safe partial DHT handoff

- Selected outbound RWI postings remain in the restart journal until every queued redundancy copy reaches a terminal local outcome. Stopping before the last copy restores the posting for later distribution rather than losing it.
- Selection commits the complete restart journal before a separate update removes live RWI rows. Restoration commits live rows before a separate update releases journal entries. Because shard files commit independently, a failed later shard can leave duplicate live and journal state, but cannot remove both recoverable copies; replay converges idempotently.
- A successful RWI response must contain `unknownURL`. An explicitly empty value completes an RWI-only handoff; an absent value is malformed and leaves the postings recoverable.
- `errorURL` from either transfer phase is applied per URL hash. Every queued sibling copy for a rejected posting is cancelled before that posting is restored, while unaffected postings continue under the final-copy rule.
- Failure to restore a rejected posting, or failure to confirm the final accepted copy in the recovery journal, produces a complete local-only retry batch. A later distributor cycle retries the local transaction before further network work, does not resend the accepted copy, and does not charge the local storage failure to a peer.
- A selected posting with authoritatively missing URL metadata is terminally finalized. A valid posting with no eligible target is restored for a later selection.
- Inbound RWI and URL metadata transfer now requires the authenticated sender hash to exist in the persistent roster. Known inactive and junior senders remain eligible; an unknown sender is rejected before storage.

### Bounded peer mail and news retention

- The YaCy message endpoint retains the wire-decoded subject and body in a durable mailbox capped at 1,024 records and 8 MiB. Hot writes maintain exact size state and evict the oldest records deterministically.
- Mailbox startup cleanup pages through keys, rejects oversized values before decoding, removes bounded confirmed-corrupt legacy records after failed decode or validation, and commits progress before continuing. An operational inspection or read failure rolls back the current page instead of treating unread state as corrupt; earlier pages remain durable, the scrub cursor remains at its last durable checkpoint, and a pending admission intent remains available for reconciliation. Peer-message permission and attachment behavior remain aligned with stock `message.java`.
- Peer news retains the incoming, processed, outgoing, and published queues under a combined 4,096-row and 4-MiB limit. The durable known-record set retains at most 4,096 identities, and out-of-order intake preserves the newest creation times.
- Ordinary news expires after 24 hours; profile-update and crawl-start records expire after 72 hours. Startup and hot-write retention remove confirmed corrupt, expired, oversized, orphaned, or category-mismatched rows without serving or publishing them. An operational inspection or read failure rolls back the current page instead of treating unread state as corrupt; earlier committed pages remain durable, cleanup resumes from its last durable cursor, and admission or rotation intents remain until replay or rollback and reconciliation complete.
- Duplicate admission and publication rotation persist bounded recovery intents before cross-shard mutation. Startup and the next write replay stable identities idempotently, repair queue sequence floors from retained physical rows, and rebuild the bounded retention view after an uncertain commit. A partial shard commit can leave temporary duplicate physical rows or at-least-once replay state until reconciliation completes; this release does not claim cross-shard transactionality.

### Exact `site:` semantics without Tavily regression

- SearchCore now carries an explicit site-host constraint. `site:example.org` matches only `example.org` and `www.example.org`, and the reverse form has the same equivalence. Arbitrary subdomains, sibling hosts, and suffix-confusion names do not match.
- The exact rule is applied consistently by local Bleve search, inbound YaCy RWI retrieval, cached-search identity, and operator-enabled web-result verification.
- Tavily `include_domains` and `exclude_domains` use separate request fields and retain parent-domain suffix behavior. Constraints are applied to local, peer, and web candidates before reciprocal-rank fusion and final truncation, with bounded overfetch and a final defensive check, so an excluded high-ranked duplicate cannot displace an allowed result. For example, an included parent domain still admits its ordinary subdomains.
- A modifier-only `site:` request remains keyword-seeded and returns the existing add-word guidance. It does not trigger an unbounded full-corpus scan.

### Test and component-boundary maintenance

- Cross-component assembly checks now live at the node assembly boundary rather than importing upper-level node behavior into lower-level component tests. Package-local persistent probes exercise mailbox and peer-news restart behavior without coupling those packages to the concrete Bolt adapter.
- A shared storage conformance suite now covers bounded bucket-key paging and encoded-value inspection across the Bolt, in-memory, and sharded vault implementations, including storage-failure behavior.
- Real sharded-bbolt regressions route generic live and journal rows to different shards, inject a failure after one actual shard commit, close and reopen storage, and verify both partial durable states. Paired production RWI state-machine regressions prove journal-before-delete, live-before-release, and idempotent recovery from equivalent partial journal and live-deletion states. File-backed component probes close and reopen through the production mailbox and peer-news constructors, verify exact resumed cursors and sequence floors, and prove second-open idempotence after partial admission or publication transitions.
- Regression suites cover v0.0.20 peer-row decoding and downgrade rewrites, legacy self rows, shared-NAT peers, direct-versus-gossip observations, cooling and promotion, bootstrap freshness and capacity, alternate-address exhaustion, exact query and seed-list forms, malformed remote rows, partition coverage, partial DHT completion, retention cleanup, and the Tavily/site-host split.
- Architecture declarations now name the `sitehost` component explicitly and keep host semantics behind the narrow search-index, web-search, and node-assembly boundaries.

## Compatibility

- No environment variable, Admin setting, route, listener, port, service, image name, package path, third-party dependency, crawler RPC field, stored document encoding, Bleve mapping, or deployment topology changes.
- Existing YaCy field names and compact seed forms are preserved. The observable changes are stricter stock-compatible parsing, bounded selection, and rejection of malformed or unauthenticated data that was previously accepted or misclassified.
- Seed hash, not address, is peer identity. Operators may therefore see a stale self row disappear after upgrade and may see distinct peers sharing one public endpoint remain listed. Cooling and junior peers stay in the known-peer roster but are excluded from reachable routing projections.
- Inbound public reachability no longer closes outbound DHT distribution by itself. Storage, network-size, peer-availability, local-RWI, distribution-policy, and activity gates remain authoritative.
- A remote peer response containing any malformed or URL-mismatched `wi` row is discarded as one invalid response. A syntactically successful transfer that omits required `unknownURL` evidence remains recoverable rather than being acknowledged.
- Search `site:` has exact bare/one-`www.` semantics. Tavily include/exclude domain arrays continue to use suffix semantics, so existing agent clients retain their documented domain behavior.
- Existing peer rows, DHT recovery records, message rows, news rows, URL metadata, RWI postings, documents, and crawler state remain readable. Startup can delete the local node's own stale peer row, expired peer rows, peers beyond the documented roster capacity, and mail/news records that exceed the documented retention, validity, or expiry bounds.
- New peer writes retain the exact v0.0.20 timestamp-plus-seed row encoding. Retry, expiry, and verification evidence live in additive internal lifecycle and bounded-cleanup-cursor keyspaces that v0.0.20 ignores. Lifecycle evidence is bound to the complete primary row and its own state; a missing, malformed, corrupted, or older-version-rewritten binding falls back to conservative legacy semantics and is removed by bounded startup cleanup.
- The packaged crawler has no behavioral or contract change in this release, but the package contains both products and normal deployment still upgrades and restarts them together.

## Upgrade and rollback

1. After the immutable tag workflow succeeds, download the architecture-matching v0.0.21 Debian package. Verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.21`, and the tagged source commit before stopping either service.
2. Follow the target's data-protection policy. Generic bare-metal nodes take one coordinated stopped backup. The documented `root@yagoseek.dev` exception preserves every existing archive under `/opt/yago/backups` and creates no new pre-upgrade backup; stage the verified package under `/opt/yago/releases`.
3. Stop `yago-crawler` and then `yago-node`. Install the package with `dpkg -i` and confirm that package installation did not start either service automatically.
4. Start `yago-node` first. Verify the exact version, `/health`, `/ready`, configured listeners, the absence of the local hash from Admin Network, peer counts and types, public-reachability evidence, DHT gates, and current logs.
5. Start `yago-crawler` after node readiness. Verify its exact version, control and ingest connections, a current heartbeat, durable queue progress, and absence of repeating lease-loss output.
6. Keep production verification observation-only. Exercise peer faults, malformed wire rows, DHT recovery, and independent process restart scenarios in the local Docker gate rather than against the production node.

For rollback, stop crawler and node, reinstall the v0.0.20 package, start and verify the node, and then start the crawler. No data restore is required solely for format compatibility. The older node reads the unchanged primary peer rows and ignores the additive lifecycle, category-evidence, recovery-intent, and cleanup-cursor keyspaces. A downgrade may leave those ignored sidecars in place or rewrite primary peer-news rows without updating them; a later v0.0.21 startup rebinds valid evidence and discards stale or orphaned state during bounded reconciliation. If v0.0.20 rewrites a peer-roster row, a later v0.0.21 start rejects the stale lifecycle binding and resumes with conservative legacy state. The older node does not restore the v0.0.21 identity, lifecycle, remote-row, multi-address, or local-completion safeguards. A crash before the last DHT acknowledgement remains at-least-once in v0.0.21 and can produce a duplicate transfer; rollback must not assume exactly-once delivery. Rows removed by bounded mailbox/news cleanup, stale-self cleanup, peer expiry, or roster-capacity enforcement are not recreated by rollback.

## Known limitations

- Outbound DHT recovery is intentionally at least once. A process stop after one redundant peer accepted a posting but before every local completion was durable can cause a later duplicate transfer; it must not cause silent loss.
- The complete remote-search response is rejected when any resource carries a malformed, incomplete, or URL-mismatched WordReference. The node does not salvage apparently valid sibling rows from a response whose integrity boundary failed.
- `prefer`, `profile`, `collection`, and `timezoneOffset` remain accepted YaCy search fields without an invented RWI-layer signal. File type, scheme, author, language, site host, and URL filtering use directly representable bounded evidence.
- Validated remote result metadata can enter the optional document/full-text cache, but a peer-attached RWI is not promoted into the local exchange vault without durable sender provenance and expiry. Peer `references` do not influence ranking or query expansion.
- A `site:` modifier without a retrieval term does not enumerate the site. Exact host filtering remains bounded to a real keyword search. Cancelling a crawl leaves already indexed documents intact; removal is a separate index operation.
- The local seed's observed type is process-local. It resets to virgin after restart and changes when fresh external hello evidence arrives; an operator can therefore briefly see an unconfirmed or virgin state after a healthy restart.
- Peer mail and news retention limits are fixed protocol and security bounds rather than runtime settings. Cleanup permanently removes corrupt, expired, oversized, orphaned, category-mismatched, or over-capacity legacy records.
- Missing, malformed, and numeric-zero peer version values retain stock YaCy developer-version eligibility under a positive seed-list floor. This compatibility rule does not treat those values as evidence of a production software release.
- The peer-news primary known-record marker remains exactly `1`, as in v0.0.20. Generation-bound category evidence, recovery intents, and cleanup cursors occupy additive keyspaces that v0.0.20 ignores.
- Mail and news recovery is intentionally idempotent rather than cross-shard transactional. A later-shard failure can leave temporary duplicate physical rows or at-least-once replay state until the next bounded reconciliation completes.
