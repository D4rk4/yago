# GitHub release notes

Each release tag has one human-authored engineering memo in this directory. The file name is the complete semantic-version tag, including its leading `v`, followed by `.md`; for example, `v0.0.9.md`.

The release workflow refuses to publish a tag without its matching memo. Begin with `## Abstract`, place the literal `<!--more-->` delimiter after a self-contained abstract of no more than 120 words, and then use these headings in order:

- `## Release identity`
- `## Changes`
- `## Compatibility`
- `## Upgrade and rollback`
- `## Verification`
- `## Known limitations`

Use precise plain English. Derive every statement from the tag-to-previous-tag diff and recorded validation evidence. State operator-visible behavior, compatibility, upgrade and rollback consequences, verification, and known limitations. Do not use an autogenerated commit list as the release notes.
