## Abstract

YaGo v0.0.17 restores crawler ingest progress when a sharded content-cluster projection is missing or no longer lists a retained URL after crash recovery or storage rollback. Replacement and deletion now use the durable URL fingerprint as bounded repair evidence: the ordinary transition reconstructs or clears membership, orphaned candidates remain invisible, full clusters rehome deterministically, and malformed records still fail closed. This removes a livelock in which one structural orphan repeatedly deferred an entire 16-document ingest group. Repair is on touch, performs no collection scan, and introduces no setting, dependency, stored-format, API, listener, service, container, or YaCy wire change.

## Changes

### Bounded content-cluster reconstruction

- Content-cluster membership remains a derived projection. A retained URL fingerprint is now authoritative point evidence when the referenced projection is missing or its visible member list omits that URL.
- Lookup treats that narrow structural absence as unpublished instead of returning an operational error. Corrupt rows, I/O failures, and cancellation remain errors.
- Candidate matching skips an orphaned fingerprint, so stale exact-content or similarity postings cannot attract another document into an unpublished cluster.
- A replacement with unchanged content reuses the retained fingerprint as the internal transition predecessor and reconstructs membership through the existing reservation and finalization protocol. Metadata changes made by the same observation are retained.
- A replacement with changed content follows ordinary candidate planning and removes the stale candidate postings. Its public result correctly reports that no previous assignment was visible.
- A deletion clears the orphaned fingerprint and postings, refreshes surviving representatives, and reports a successful deletion without inventing a previous visible assignment.

### Capacity and batch isolation

- Recovery reuses the prior cluster only when its visible members plus earlier plans in the same batch remain below the configured member limit.
- A full cluster selects a deterministic recovery identity derived from the URL, content identity, and former cluster identity. Replaying the observation therefore converges on the same cluster without overflow.
- Completion verifies that the committed cluster actually publishes the current URL before returning the assignment.
- One recoverable orphan no longer aborts the other rows in a bounded 16-document ingest group. The work remains limited to the touched fingerprint, bounded candidate postings, and affected cluster members; no startup or request-time collection scan was added.

### Failure model and documentation

- The content-cluster fault engine can model complete loss of one physical shard while other shard state survives. Regression coverage exercises the same split fingerprint/projection failure observed in production.
- The fleet-rate fencing regression now asserts the durable termination and disconnected state of both crawler sessions instead of the scheduler-dependent number of process-smoother directives queued after cancellation. The old assertion reproduced intermittently in both normal and race runs; production fencing and ordering are unchanged.
- README, the normative specification, YagoRank documentation, the sharded-vault and lineage-deletion ADRs, and the feature catalog now describe on-touch reconstruction and its limits.
- No configuration knob is necessary: the behavior repairs an invariant and applies automatically to every replacement and deletion.

## Compatibility

- The release is a drop-in upgrade from `v0.0.16`; node and crawler should still be upgraded together from the same package.
- Vault keys, encoded rows, Bleve indexes, crawler checkpoints, node-crawler protobufs, Admin routes, public search APIs, Tavily-compatible responses, and YaCy wire endpoints are unchanged.
- Existing healthy clusters retain their identifiers and no-op replacement behavior.
- Missing or member-omitting projections remain invisible until the affected URL is replaced or deleted. That operation uses existing records and reservations; it requires no migration or reindex.
- A full surviving cluster may place the repaired URL in a deterministic new cluster rather than exceeding the existing member bound.
- There is no new environment variable or Admin setting, and deployment examples require no change.

## Upgrade and rollback

1. Download the architecture-matching published v0.0.17 package and verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.17`, and the tagged source commit.
2. Back up the complete node and crawler data directories while both services are stopped. Preserve existing archives separately from the release staging directory.
3. Stop `yago-crawler`, then stop `yago-node`. If the node was stopped earlier, confirm it remains inactive and stop the crawler before package installation.
4. Install the matched package containing both binaries with `dpkg -i`. Package installation enables but does not start the services.
5. Start `yago-node` first and verify `/health` and `/ready`. Start `yago-crawler` only after node readiness.
6. Verify one stable crawler session, advancing document totals, an empty ingest-defer loop, and the intended public, Admin, peer, and crawler listener states.

Rollback to `v0.0.16` does not require a format conversion because this release changes no stored encoding. Use a coordinated stopped backup so document, search, cluster, broker, and crawler-checkpoint state remain from one boundary; install both older binaries, start the node, verify readiness, and then start the crawler.

## Known limitations

- Reconstruction is deliberately on touch. It does not scan all clusters during startup, so an orphan remains unpublished until its URL is replaced or deleted.
- Repair requires the retained URL fingerprint. If both fingerprint and projection evidence are absent, a later crawl is admitted as new evidence rather than reconstructing an unknown former assignment.
- A malformed fingerprint, cluster row, posting, or transition remains an error. The repair handles structural absence, not arbitrary corruption.
- Cross-shard vault atomicity remains relaxed. Reservations, transitions, visibility markers, idempotent replay, and this point repair bound failure; they do not create one transaction across every shard, Bleve, document storage, anchors, RWI, and URL metadata.
- The crawler remains at least once. An in-flight page whose durable outcome did not commit may be fetched again after restart.
- bbolt still permits one read-write transaction per database. Sharding, bounded ingest groups, and on-touch reconstruction reduce contention and retry amplification; they do not eliminate storage latency or repair an underlying host I/O failure.
