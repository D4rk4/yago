## Abstract

YaGo v0.0.13 bounds crawler control-plane work so a busy node cannot stream leases faster than a crawler can durably accept them. Ordinary delivery now uses one confirmed lease credit per session; reconnect recovery advances in confirmed batches of at most 16. Targeted heartbeats, constant-time lease-capacity accounting, cancellable database-writer admission, and settlement-aware grant handling prevent control RPCs from queuing behind stale work. Operators should upgrade node and crawler together. The release changes no stored schema, protobuf shape, listener, environment variable, search API, or Admin UI surface, so it requires no migration or reindex.

## Changes

### Bounded order delivery

- Each ordinary streamed order consumes one session-scoped delivery credit after its durable lease claim. The node registers the expected lease before sending it and neither claims nor sends the next order for that session until a successful heartbeat renews the lease.
- A successful session-authorized ACK, NAK, legacy termination, or rich terminal disposition of the exact lease can also satisfy its pending credit. This preserves progress when a compatible crawler settles a delivery before a confirmation heartbeat completes.
- The current crawler sends the targeted confirmation before decoding an ordinary payload. An undecodable payload therefore remains represented by a known lease, and a failed malformed-order settlement terminates the stream attempt so reconnect recovery can replay it instead of leaving delivery credit parked.
- Delivery-credit waits hold neither a bbolt transaction nor the worker-session registry lock. Independent crawler sessions remain able to receive work while one session waits for confirmation.

### Bounded reconnect recovery

- One worker session may still retain at most 1,024 active leases, but reconnect recovery divides them into ordered batches of at most 16.
- The first frame carries the current batch's complete lease-ID header. The node waits for one targeted heartbeat to renew every lease in that header before it sends the remaining frames, and the crawler confirms the batch before exposing its first recovered order.
- Recovered payloads remain streamed and are not accumulated as a complete recovery set. The crawler validates every payload against the header and requires the exact batch-final marker.
- Periodic heartbeats continue to carry the complete active lease set. Ordinary and recovery-confirmation heartbeats carry only the lease or batch being admitted, reducing repeated lease work without weakening periodic loss detection.

### Lease accounting and storage admission

- Crawl-broker startup rebuilds an in-memory active-lease catalog from the durable lease bucket. Capacity checks use one worker/session lookup instead of scanning every lease for every streamed order.
- Successful claim, adoption, settlement, defer, and requeue paths update the catalog only after the durable transaction commits. The persistent lease bucket remains authoritative across restart, and failed transactions do not publish speculative catalog state.
- The dedicated bbolt engine serializes write admission with a context-aware token. A cancelled RPC can stop while queued; an admitted update checks cancellation again before its transaction callback and before commit, causing stale work to roll back rather than enter a later writer convoy.
- Provisioning without a request context retains unbounded admission. Read transactions retain their existing concurrency behavior.

### Progress and settlement correctness

- A running progress report now authorizes its exact lease, worker, process session, and run once, then reuses that verified run target for control reconciliation and recording. The previous second full-lease scan is removed.
- Human-facing crawler progress warnings encode byte-backed run identities as lowercase hexadecimal text.
- Settlement attempts use reference-counted grant protection. A grant remains renewable and cannot expire or become an omission loss while an ordinary or rich terminal settlement is in flight, including when a heartbeat captured the lease before the node committed the settlement.
- If every settlement attempt fails before durable completion, normal omission and expiry handling resumes immediately. Successful ordinary settlement removes the grant without signaling lease loss. Rich terminal settlement removes it after its first-phase token is durable and before final confirmation, so neither successful path reconnects an otherwise healthy order stream.

## Compatibility

- YaCy `/yacy/*`, public search, OpenSearch, Tavily-compatible APIs, ranking, crawler HTTP behavior, and Admin UI are unchanged.
- No protobuf message or RPC shape changes. Targeted confirmation and complete periodic renewal use the existing bounded repeated lease-ID field. Stock older decoders therefore see no new field.
- A current crawler accepts the v0.0.12 single recovery-batch shape up to the existing 1,024-lease ceiling, and unmarked recovery retains per-order confirmation. Supported operation nevertheless upgrades `yago-node` and `yago-crawler` as a matched v0.0.13 pair because delivery-credit and intentional-settlement behavior spans both processes.
- No stored schema, database path, port, listener, service name, container name, setting, or environment variable changes. Existing `crawlbroker.db` and crawler frontier files open in place. No migration, frontier reset, document reindex, or Bleve rebuild is required by this release.
- The crawler remains at-least-once. A response that began but whose durable page outcome did not commit may still be fetched again after failure.

## Upgrade and rollback

1. Download the architecture-matching published v0.0.13 package and verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.13`, and the tagged source commit.
2. On a generic bare-metal installation, back up the complete node and crawler data directories before stopping services.
3. Stop `yago-crawler`, then stop `yago-node`. Install the matched package containing both v0.0.13 binaries with `dpkg -i`; package installation enables but does not start the services.
4. Start `yago-node` first and verify `/health` and `/ready`. Start `yago-crawler` only after node readiness, then verify one stable worker session, bounded active leases, progressing crawler tallies, and the absence of repeated heartbeat or settlement timeouts.

For `root@yagoseek.dev`, the authorized production exception creates no new pre-upgrade backup. Preserve every existing archive under `/opt/yago/backups` unchanged, stage and verify the published package under `/opt/yago/releases/v0.0.13`, stop both services, install it, then manually start and verify the node before the crawler. This exception accepts reduced rollback coverage and does not authorize opening, replacing, pruning, or recreating the live persistent databases.

The durable formats are unchanged. To roll back, stop crawler before node, install the matched published v0.0.12 package, start and verify the node, then start the crawler. Restore a matching pre-upgrade backup only when a point-in-time rollback is required; doing so discards crawler and index progress committed after that backup.

## Known limitations

- Delivery credit is scoped to one worker process session. It bounds intake for that session rather than imposing a fleet-wide crawl or pages-per-second limit; aggregate throughput still depends on configured crawler concurrency, active sessions, per-run rates, and host politeness.
- One session can retain at most 1,024 active leases, while recovery admits batches of at most 16. These are fixed liveness bounds rather than Admin settings in this release.
- bbolt continues to permit one read-write transaction at a time. Context-aware admission prevents cancelled RPC work from joining a stale writer convoy; it does not make durable writes parallel or eliminate storage latency.
- The crawler provides at-least-once execution. A page whose response began but whose durable outcome did not commit before termination may be fetched again once.
- A final settlement transport failure can cause reconnect and idempotent replay even when the remote outcome is not yet known. Lease tokens and durable settlement history prevent a stale process from completing a different active lease.
- Recovery state remains local to the configured node and crawler data directories. This release adds no cross-host frontier replication, distributed lease catalog, or fleet-wide checkpoint protocol.
- This maintenance release does not change search recall, morphology, ranking, PDF extraction, browser rendering, Tavily compatibility, or Admin UI behavior beyond the capabilities and limitations already documented for v0.0.12.
