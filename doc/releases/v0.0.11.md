## Abstract

YaGo v0.0.11 makes crawler work durable across process and node restarts, moves crawl coordination into one atomic database, and renames the worker to `yago-crawler`. It aligns all four Tavily-compatible endpoints, strengthens analyzer evidence, bounded morphology, mixed-source learned ranking, CJK analysis, public paging, and PDF simple-font extraction. Reserved-free storage policies now pause gate-managed node and crawler growth before filesystem exhaustion, while bounded API-key and session retention limits administrative state. Operators must upgrade node and crawler together, provision persistent crawler data, migrate legacy service configuration, allow Bleve shards to rebuild, and provision a filesystem or project quota when a hard aggregate storage limit is required. Listeners and ordinary YaCy wire behavior remain compatible.

## Changes

### Persistent crawler frontier

- The crawler stores run definitions, ordered outstanding pages, visited URLs, page ownership, host availability and pace, the complete expanded-seed manifest, controls, tallies, and terminal-delivery state in bbolt under its configured `YAGO_DATA_DIR`.
- A restart recovers no more than 256 resident pages per run and refills below 128 from durable cursors. A large frontier therefore resumes without loading or reparsing the complete seed set.
- Stable worker identity, per-process session identity, immutable lease tokens, mutation-time authorization, and bounded same-worker lease adoption fence stale processes after crawler or node restart.
- One recovered heartbeat confirms as many as 1,024 adopted leases before their payloads are replayed. Older protobuf decoders continue to accept the additive fields.
- Page outcome, run tally, host progress, visited membership, and frontier removal commit atomically. A committed page is not fetched again; work whose response began but did not commit remains eligible for one at-least-once replay.
- Pause, resume, cancellation, per-run rate changes, automatic-discovery priority, host retirement, and terminal settlement remain durable across reconnect and restart.
- Run lifecycle reporting publishes the initial `running` state with its seeded queue depth before a fast worker can publish the terminal state; periodic snapshots continue with the live pending depth.
- Every new manual, scheduled, swarm-discovery, and web-discovery order carries an additive whole-run page budget in its crawl profile. The node and crawler bootstrap from `YAGO_CRAWLER_MAX_PAGES_PER_RUN` (50,000 by default), Configuration → Crawler changes the default live for subsequent tasks, the manual form can override one task, `0` is unlimited, and recrawls retain the original profile value.

### Bounded browser escalation

- A successful fast-path HTML response with executable JavaScript now enters the existing Firefox fallback when bounded main-text extraction finds fewer than four terms and fewer than sixteen Unicode letters or digits. This recovers client-rendered application shells that previously indexed their empty initial document.
- Usable static HTML remains a single HTTP fetch. Scriptless and non-HTML responses, empty or data-only scripts, and crawl profiles that disable browser rendering do not escalate. The existing browser pool, acquisition deadline, failure threshold, cooldown, body limit, and public-web guard remain authoritative.

### Atomic node crawl coordination

- `${YAGO_DATA_DIR}/crawlbroker.db` is the single bbolt transaction domain for pending and leased orders, lease and worker-session state, control delivery, settlement history, and terminal run delivery.
- First startup performs bounded, resumable copies of the frozen version-1 broker and terminal-run legacy bucket sets. Source and destination fingerprints and row totals must agree before their independent completion markers are committed.
- Legacy source rows remain unchanged. A migration conflict or unverifiable partial target fails startup instead of silently mixing states.
- Opening the crawl database has a five-second lock bound. Failed assembly closes partially opened state before returning.
- Prometheus reports logical state bytes and physical crawl-database bytes. This dedicated database is outside the main sharded-vault quota and compaction path and has no independent byte cap in this release.

### Filesystem pressure and bounded administration

- The node and crawler have independent reserved-free and recovery-hysteresis policies because their data directories may use different filesystems. The bootstrap variables are `YAGO_STORAGE_RESERVED_FREE=1GB`, `YAGO_STORAGE_PRESSURE_HYSTERESIS=256MB`, `YAGO_CRAWLER_STORAGE_RESERVED_FREE=1GB`, and `YAGO_CRAWLER_STORAGE_PRESSURE_HYSTERESIS=256MB`; Configuration exposes matching live controls.
- Each process measures available blocks on the filesystem containing its own data directory. Measurement failure fails closed. Ordinary checks share one observation for at most one second, and pressure remains active until free space reaches the reserve plus hysteresis.
- Gate-managed node crawl/index ingestion, new crawl orders, and connected best-effort writes pause under pressure while removal, eviction, deletion, settlement, and recovery paths remain available. Crawler workers retain their exact checkpoint and wait before new fetch and frontier-growth admission.
- Compaction and shard splitting measure their actual copy source and share one maintenance lock across a forced fresh filesystem check and the complete copy. Each 256-row retained-state migration page reserves its payload plus bounded bbolt allocation headroom. A legacy single-file vault migration preflights its source size.
- Initial creation of a missing or zero-length `crawlbroker.db` reserves 1 MiB. A valid existing crawl database opens without pressure admission so recovery and cleanup startup remain possible. Missing legacy source buckets are treated as empty without mutating the retained source; storage engines that cannot report bucket presence fail startup.
- Worker heartbeats carry optional storage availability, pressure, measurement status, and active fetches. Storage and utilization observations expire after three heartbeat intervals; older, omitted, invalid, or stale reports render as unreported or unavailable rather than preserving obsolete state.
- Node and crawler Prometheus series expose available bytes, reserve, hysteresis, pressure, measurement availability, rejected growth, and measurement failures. Admin storage labels distinguish logical live main-vault data and its soft admission target from physical filesystem allocation.
- The durable API-key store admits at most 256 new keys. Creation reports HTTP 409 at capacity; last-used telemetry is best effort, coalesced to at most once per key per five minutes, and bounded for legacy over-cap stores. Authorization uses 32 nonblocking process slots and reports HTTP 503 with `Retry-After` when saturated.
- Raw API-key listing now uses native bounded keyset pages. `cursor` is exclusive, `limit` defaults to and cannot exceed 256, and continuation responses expose `nextCursor`, `truncated`, and `total`; a default request against at most 256 keys preserves the prior keys-only response shape. Legacy over-cap stores remain listable and revocable page by page.
- Admin → Security renders 20 API keys per page with Previous/Next links, bounded validated cursor history, and a recovery link when a stale cursor reaches an empty page.
- Cookie authentication retains at most 256 active sessions. Session creation removes expired legacy rows in bounded pages and deterministically evicts the oldest remaining session at capacity.

### Bounded PDF simple-font extraction

- Embedded `/ToUnicode` mappings retain precedence for every code they define. A missing mapping in a simple font may use a bounded single-byte decoder instead of interpreting the stored glyph code as text.
- The decoder resolves `/Encoding` as a predefined name or an inline or indirect dictionary, initializes a private 256-code table from a supported `/BaseEncoding`, applies `/Differences` in order, and derives Unicode text through standard glyph-name rules.
- Unknown glyph names, malformed arrays, unresolved or cyclic references, invalid Unicode scalar names, and exhausted work budgets leave affected codes unmapped. A selected custom font therefore fails closed instead of producing plausible ASCII, Latin-1, or heuristic ligature garbage.
- A bounded synthetic fixture preserves the reported Cisco ENCS custom-encoding structure without redistributing the external document. The live Cisco file remains the required verification case, and an existing cached or indexed copy requires one normal recrawl after upgrade. Extraction remains embedded-text only; no OCR is added.

### Document format routing

- The Text format family explicitly admits `.md` and `.markdown` URLs, including servers that label them `application/octet-stream`, and extensionless `text/markdown` or `text/x-markdown` responses. Binary-looking bodies remain unparsed.

### Search evidence and morphology

- Local Bleve rows retain bounded absolute match positions from stored document bodies. The existing `/cached` route can render an escaped, UTF-8-safe contextual passage for those positions while ordinary cached-copy URLs remain compatible.
- Peer, web, and legacy rows receive analyzer-backed evidence over a bounded 500-row visible window. Structural matching remains only the fallback when analyzer evidence is invalid, unavailable, or deadline-limited.
- Multiword morphology unions bounded sibling forms inside each query requirement and intersects requirements. It does not form a Cartesian product or replace the primary exact RWI request.
- Swarm morphology remains operator-controlled and disabled by default.
- Cooperating Yago peers can negotiate one bounded analyzer-recall request and version-1 query-bound evidence. Exact query hashes, requirement ordinals, and URL allowlists prevent unrelated remote evidence from being accepted.
- Federated evidence survives retained result paging while large ranking-only position maps are discarded after scoring.
- Equivalent-analyzer query plans are shared, reducing repeated local candidate scans without changing field boosts or score scale.
- An active learned model now evaluates the same bounded leading fused window across local, peer, and web rows. Explicit missing-value semantics remain unchanged, rows beyond the window are untouched, and global serving no longer doubles its candidate request solely to collect a local-only model quota.
- Admin YagoRank now includes local and global Search Explain. The global diagnostic shares the serving retrieval, recovery, analyzer-evidence, and fusion path under the interactive deadline, but does not create a paging session, cache remote rows, or seed crawl work. It reports human-facing `local`, `peer`, and `web` sources, retrieval and final scores, reciprocal-rank branch contributions, partial failures, learned signals, and tree paths. Admin Search preserves query and scope when opening the diagnostic.

### CJK analysis

- Chinese, Japanese, and Korean indexing includes source-offset unigrams and overlapping bigrams.
- Chinese and Japanese indexing adds bounded terms derived from pinned Jieba and SudachiDict Small data through compact vellum lexicons. Query-side gates retain the mandatory character evidence so unavailable or ambiguous dictionary terms do not suppress recall.
- Equal-width Traditional-to-Simplified Chinese mappings preserve source offsets. Unequal-width conversion is deliberately excluded.
- Release packages, tarballs, and the node image install the dictionary license and provenance notice.
- The changed mapping identity rebuilds all eight existing Bleve shards from the document store before search readiness.

### Search availability and public paging

- Up to four admitted public searches may wait for an execution slot inside the existing 1.8-second response boundary instead of immediately returning an overload miss.
- A capacity-only local exact retry receives a bounded 500-millisecond recovery window. Completed local, peer, or web rows are preserved when a different source reaches its deadline.
- Search logs record bounded partial-failure sources without expanding unbounded error attributes.
- An explicitly requested public page remains selected while it is materialized. The portal returns to the last page only after completed retrieval proves the requested page is beyond the result set.

### Tavily-compatible agent API

- Default search results now retain `raw_content:null`, omit Yago-only source provenance, emit publication dates only for news, and follow the requested image and favicon shape. Every error body contains only `detail.error`.
- Search validation now follows the reference values and bounds for topic, country, domains, chunks, result limits, and safe-search combinations. `basic`, `fast`, and `ultra-fast` use the local plan; `advanced` uses global local-plus-peer retrieval.
- With click-exposure randomization disabled, equivalent root-portal and Tavily `advanced` requests preserve the same canonical URL order after cluster deduplication. Equivalent local-depth requests match the YaCy local surface because the root portal has no local mode.
- Extract, crawl, and map now accept their reference request fields, defaults, and ranges. They retain YaGo's stricter process-wide four-slot, 30-second, 16 MiB, and 200-page safety bounds.
- Extract queries and crawl instructions can select bounded lexical chunks. Answers remain deterministic extraction; instructions do not guide traversal, and map instructions do not alter discovery.

### Backup, packaging, and deployment

- Coordinated backup and restore cover the node data directory and every crawler checkpoint directory, including the dedicated crawl database and terminal outboxes.
- The canonical crawler module, command, binary, systemd and Compose service, container image, and GHCR repository are named `yago-crawler`; its environment variables use the `YAGO_CRAWLER_` prefix.
- Package and tar installers stop and disable the legacy `yagocrawler.service`, remove its unit and binary, and migrate `/opt/yago/etc/yagocrawler.env` only when the canonical environment file is absent. Migrated line-leading `YAGOCRAWLER_*` keys become `YAGO_CRAWLER_*`; an existing canonical file wins on an interrupted or repeated installation. No legacy service alias remains.
- Docker deployments use a separate persistent crawler volume; systemd and package deployments keep runtime state below the configured data directory.
- Compose now provisions the separate persistent `yago-crawler-data` volume. v0.0.10 had no crawler frontier volume to retain; durable frontier state begins when v0.0.11 first starts and survives later Compose restarts and upgrades.
- Release packages include the CJK notice. The tag workflow builds, smoke-tests, scans, attests, and publishes architecture-specific packages and exact validated multi-architecture container manifests.
- After publishing the immutable container manifests, the release workflow verifies their tag, source revision, workflow identity, architecture, OCI labels, containerized binary versions, attestations, and anonymous tag and digest pulls. A failed post-publication check blocks GitHub Release creation and release completion.

## Compatibility

- The crawler identity rename is an operationally breaking change. Current binaries, services, images, modules, and environment variables use `yago-crawler` and `YAGO_CRAWLER_*`; the legacy names are migration inputs only and are not runtime aliases.
- Ordinary YaCy `/yacy/*`, public search, and OpenSearch shapes remain unchanged. Tavily success payloads now remove the Yago-only `source` field, retain `raw_content:null` by default, omit `published_date` outside news, normalize news dates, and use the requested image shape. Errors now contain only `detail.error`; clients that depended on removed Yago-only provenance or error fields must update.
- Query-match evidence and analyzer recall use optional additive Yago fields. Stock YaCy peers ignore them and continue to answer exact RWI surface hashes.
- The crawler protobuf additions are decoder-compatible, but current session fencing requires a matching v0.0.11 node and crawler for supported operation. Upgrade both binaries together.
- Browser-enabled profiles may perform one additional bounded fetch for a successful but structurally empty JavaScript HTML response. Browser-disabled profiles retain the previous fast-path result.
- The crawl-order JSON profile adds optional `MaxPagesPerRun`. Older decoders ignore it; current crawlers use their environment bootstrap only when an older queued profile omits it. Existing active and recrawl profiles do not acquire a later runtime default.
- No port or listener changes. The crawler's `YAGO_DATA_DIR` must be persistent and preserved; Docker now supplies it through a dedicated volume.
- Existing legacy node crawl rows migrate once into `crawlbroker.db`. The retained legacy source is not a live rollback replica after v0.0.11 accepts new crawl work.
- Existing Bleve shards rebuild because the CJK mapping changed. Search readiness may take longer during the first start after upgrade.
- The four storage-pressure environment variables are new bootstrap defaults with matching live Admin controls. Current crawlers accept the policy and report their state through additive optional heartbeat fields. Those fields remain decoder-compatible, but session fencing rejects an older live crawler; supported operation requires a matching v0.0.11 node and crawler.
- A required Bleve rebuild, legacy vault copy, retained crawl-state migration page, or first crawl-database creation now fails closed when its measured headroom cannot preserve the configured node reserve. Opening a valid existing crawl database remains available under pressure.
- API-key list clients that can encounter more than 256 legacy records must follow `nextCursor` while `truncated` is true. Existing unpaged clients retain the earlier response shape when the complete store fits in one default page.
- A node with an active learned model can change mixed-source result order because peer and web rows in the leading fused window now participate in the same bounded inference pass as local rows. No additional peer or web request is introduced.
- PDF API and storage shapes are unchanged. PDFs whose text was stored by the previous byte fallback, including the reported Cisco ENCS document, retain that old text until they are recrawled.
- New direct uses are crawler-side bbolt 1.4.3 and node-side vellum 1.2.0 and snowballstem 0.9.0. The node already pinned bbolt, while vellum and snowballstem were transitive dependencies. Jieba, SudachiDict Small/UniDic, and OpenCC data are recorded in ADRs and the distributed notices.

## Upgrade and rollback

1. Download the architecture-matching published package and verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.11`, and the tagged source commit.
2. On a generic v0.0.10 installation, use that release's node-only backup tool before the upgrade. The v0.0.10 crawler frontier is volatile and is not present in that archive, so it cannot resume in-flight crawler work after the upgrade. The coordinated node-and-crawler backup format described by this release becomes available only after v0.0.11 is installed.
3. Explicitly stop the installed crawler service, including `yagocrawler.service` on a legacy installation, and then stop the node before invoking `dpkg`. Install the matched package containing both v0.0.11 binaries. Its pre-install migration is the service-name boundary: it removes the offline legacy unit and binary and conditionally migrates the environment file. Never restart the legacy service after package installation.
4. Start the node first. Wait for the crawl-state migration, Bleve rebuild, `/health`, and `/ready` before starting the crawler.
5. Start `yago-crawler` and verify that it connects and persists newly dispatched work. A v0.0.10 crawler has no durable frontier or tallies to resume; restart continuity applies after v0.0.11 has created its frontier. Compose and manually managed deployments must rename any remaining `YAGOCRAWLER_*` inputs to `YAGO_CRAWLER_*`; only package-managed environment files are rewritten automatically.
6. Verify local, peer, and operator-enabled web search, a second public result page, and crawler progress before restoring traffic.
7. Schedule a normal recrawl for PDFs whose cached or indexed text was produced by an older extractor; include the reported Cisco ENCS URL in production verification.

Before the first v0.0.11 start, verify that each node and crawler data
filesystem has enough free space above its configured reserve for the Bleve
rebuild and any legacy copy. If startup rejects growth, free filesystem space or
temporarily lower the applicable reserve and hysteresis, then retry. Admin
cleanup is unavailable until the node starts. Deleting bbolt rows can make pages
reusable inside a database without increasing operating-system free space, so
filesystem cleanup or compaction may still be required.

For `root@yagoseek.dev`, the authorized production exception creates no new pre-upgrade backup. Existing archives under `/opt/yago/backups` remain unchanged. The attested package is staged under `/opt/yago/releases/v0.0.11`; the crawler is stopped before the node; the package is installed; the node is started and verified before the crawler. This accepts reduced rollback coverage.

To roll back to v0.0.10, stop crawler before node, restore the matching v0.0.10 node-only pre-upgrade backup, install the older matched node and crawler package, start and verify the node, then start the crawler. There is no v0.0.10 crawler-frontier archive to restore. Do not open a v0.0.11 frontier with an older crawler. Without the matching node backup, retained legacy broker rows are stale and can resurrect pre-migration work; all post-upgrade crawler progress is lost on this rollback.

## Known limitations

- A page whose response began but whose durable outcome did not commit before termination may be fetched again once. The crawler provides at-least-once execution, not transactional control of the remote origin.
- JavaScript-shell selection is a bounded structural text-quantity decision, not semantic understanding. A very short script-enhanced page can render unnecessarily, while a shell containing enough placeholder text can remain on the fast path.
- Checkpoint recovery requires the same or a transferred crawler `YAGO_DATA_DIR`; this release does not provide transparent cross-host frontier replication.
- Crawl rates remain bounded per run, per host, and per crawler process. There is no fleet-wide aggregate pages-per-second ceiling.
- `YAGO_STORAGE_QUOTA` remains a soft target for logical live main-vault rows. It excludes Bleve, the node crawl database, crawler frontiers, bbolt free pages, open-but-deleted blocks, and temporary compaction, split, migration, and merge copies.
- Reserved-free policies are advisory admission controls for gate-managed growth, not atomic aggregate quotas. Ordinary writes can allocate after preflight; bounded bootstrap writes and storage-engine background work remain outside one cross-process transaction. A hard maximum requires an operator-provisioned filesystem or project quota, or a quota-capable volume, for every data placement.
- The dedicated crawl database has no independent byte cap. Deleting bbolt rows may leave reusable pages without returning blocks to the filesystem; monitor logical and physical database metrics plus filesystem free space. A pressure-blocked startup Bleve rebuild requires out-of-band filesystem cleanup or a policy adjustment before Admin cleanup becomes available.
- Legacy broker states that were already internally inconsistent are copied exactly; migration cannot reconstruct an order or index row that never existed. Visible source/target conflicts fail startup.
- Sustained search saturation can still return a bounded partial or unavailable response inside the 1.8-second deadline. An enabled external web provider is not guaranteed to answer.
- Morphology is bounded and analyzer-dependent. Irregular or unsupported forms may remain unexpanded, and stock YaCy peers continue to retrieve exact surface hashes only.
- Federated analyzer evidence is query-bound and session-retained, not a persistent cross-node offset database. Invalid or deadline-limited evidence uses bounded visible-field fallback.
- CJK analysis does not provide complete word-sense disambiguation, Japanese readings, unequal-width script conversion, or Korean dictionary segmentation.
- Public result sessions retain at most 500 rows for five minutes. Navigation cannot expose a page outside that materialized window.
- Populated Bleve shards commit concurrently but not atomically across shards if one backend commit fails.
- PDF extraction remains limited to embedded text. Composite fonts without a usable Unicode map and simple-font glyphs without a trusted standard mapping remain unparsed, and this release performs no OCR.
- Tavily-compatible answers are deterministic extractive text, not model-generated answers. `auto_parameters` performs no intent inference; country does not add geographic ranking; finance has no dedicated vertical; extract depth uses one extraction engine; basic, fast, and ultra-fast share one local retrieval plan.
- Tavily-compatible query and instruction chunks use bounded lexical selection. The node has no proprietary semantic reranker, model-guided traversal, upstream Tavily provider, image-ranking model, or real credit accounting.
