## Abstract

YaGo v0.0.14 corrects crawler control-plane starvation discovered when v0.0.13 failed production acceptance. It snapshots ingest authorization before slow storage, bounds each crawler to a live default of 32 active tasks independently of fetch workers, keeps progress delivery fair, and retains every active run record. Reconnect recovery now declares one adopted-lease manifest and separates keepalive from exact delivery confirmation, preventing periodic heartbeats from releasing unseen batches. The protocol additions are protobuf-compatible; operators upgrade node and crawler together, while the shared bootstrap setting defaults to 32 when omitted. No database migration, reindex, listener, or YaCy wire change is required.

## Changes

### Snapshot ingest authorization

- The node validates the current worker, process session, exact lease, and run under the short lease-mutation boundary before any ingest storage side effect.
- It releases that boundary before document storage, Bleve indexing, URL metadata, RWI postings, stale-state removal, or recrawl updates. A long ingest microbatch therefore cannot retain the lease read fence while storage completes.
- A failed snapshot is rejected before its first storage operation. A successful snapshot is the linearization point for the existing idempotent absorption; later lease movement does not cancel already authorized storage work.

### Active-task workbench

- Each crawler process now admits a bounded number of distinct active crawl tasks. `YAGO_CRAWLER_MAX_ACTIVE_RUNS` and the live Admin key `crawler.max_active_runs` default to 32 and accept values from 1 through 256.
- An active-task slot spans prepared-order admission through terminal completion. Additional ordinary and recovered orders wait without activating another frontier or periodic progress reporter.
- Increasing the live value wakes waiting orders. Decreasing it does not cancel or preempt active tasks; replacement admission resumes after occupancy falls below the new value.
- Active-task admission is independent of `YAGO_CRAWLER_WORKERS` / `crawler.fetch_workers`. Fetch workers continue to bound page-fetch jobs only.

### Recovery manifest and delivery credit

- The first recovered frame declares the complete adopted-session lease manifest once. The node retains the 1,024-lease session ceiling and streams ordered batches of at most 16.
- Every later batch must be a nonrepeating subset of the declared manifest. The crawler rejects undeclared or repeated leases, an incomplete recovery prefix, recovery after ordinary streaming starts, and ordinary delivery before manifest completion.
- `WorkerHeartbeat.confirm_active_lease_deliveries` has optional presence. Periodic full-set heartbeats send false and renew leases without releasing unseen delivery credit. A targeted true confirmation must renew exactly the currently expected ordinary lease or recovery batch.
- An absent marker preserves the v0.0.13 subset-confirmation rule for an older crawler. Current crawlers retain the legacy unmarked and single-batch recovery paths when connected to an older node, although supported deployment still uses matched binaries.

### Progress and run-state bounds

- The crawler progress queue keeps an in-flight generation immutable. Later running snapshots for that run coalesce into one pending replacement; ready terminal reports retain priority, and another ready run is serviced before the replacement.
- The bounded queue may omit intermediate running values without collapsing a protected `terminal → running → terminal` phase chain or evicting the in-flight attempt.
- The node keeps one successful worker assignment per active run. Repeat running reports from the same worker avoid persistent directive reconciliation; worker migration reconciles before replacing the assignment, and successful completion removes it. Run-targeted Admin directives resolve through that authoritative assignment, so pause, resume, cancel, and rate controls cannot be queued for a stale worker after migration.
- The node run registry retains every active run and updates the active total without a full scan. Its configured capacity now bounds only recent terminal history, whose eviction is deterministic by update time and run identity.

### Contract and configuration surface

- `CrawlOrderMessage.recovered_session_lease_ids` is additive protobuf field 6.
- `WorkerHeartbeat.confirm_active_lease_deliveries` is additive optional protobuf field 9.
- `CRAWL_CONTROL_KIND_SET_ACTIVE_RUNS` is additive enum value 8, and `CrawlControlDirective.maximum_active_runs` is additive field 7.
- Docker Compose and both systemd environment examples carry `YAGO_CRAWLER_MAX_ACTIVE_RUNS=32` for node and crawler bootstrap parity. Configuration → Crawler exposes the live value as “Maximum active crawl tasks.”

### Admin operator surfaces

- System Monitor keeps the connected crawler fetch meter and crawler storage reserve alongside the independent node filesystem reserve. When the crawler disconnects, only crawler-owned readings disappear; the node reserve does not replace them.
- Search Explain no longer inherits the narrow settings-form width. Its six-column results table uses the available desktop width and remains reachable through a focusable horizontal scroller on narrow viewports.
- Configuration → Crawler renders the new active-task value as 32 by default, accepts the inclusive 1–256 range, rejects values outside it, and resets to the bootstrap default without disturbing adjacent row geometry.

## Compatibility

- YaCy `/yacy/*`, public search, OpenSearch, Tavily-compatible APIs, ranking, document formats, and crawler page-fetch behavior are unchanged.
- The protobuf additions preserve existing field numbers. Older decoders ignore fields 6, 7, and 9 and enum value 8; an absent confirmation marker deliberately retains legacy subset confirmation.
- Supported operation upgrades `yago-node` and `yago-crawler` together. Decoder compatibility is not a guarantee that a mixed v0.0.13/v0.0.14 control plane has the corrected capacity and confirmation semantics on both sides.
- Existing `crawlbroker.db`, crawler frontier checkpoints, terminal state, and search indexes open in place. No database migration, frontier reset, document reindex, or Bleve rebuild is required.
- The new environment variable is a bootstrap default with a matching live Admin setting. Operators using managed environment files should set the same value for node and crawler; the default is 32 when it is omitted.
- The ingest authorization snapshot changes lock duration, not ingest identity or durability. Existing observation IDs, newest-observation ordering, and at-least-once retry behavior remain authoritative.

## Upgrade and rollback

1. Download the architecture-matching published v0.0.14 package and verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.14`, and the tagged source commit.
2. Set `YAGO_CRAWLER_MAX_ACTIVE_RUNS` consistently in the node and crawler environment files when an explicit bootstrap is required. Omitting it selects 32 in both binaries.
3. On a generic bare-metal installation, back up the complete node and crawler data directories before stopping services.
4. Stop `yago-crawler`, then stop `yago-node`. Install the matched package containing both v0.0.14 binaries with `dpkg -i`; package installation enables but does not start the services.
5. Start `yago-node` first and verify `/health` and `/ready`. Start `yago-crawler` only after node readiness. Verify one stable worker session, active-task occupancy at or below the configured limit after recovery settles, advancing progress, and no repeated heartbeat, progress, or settlement timeout wave.

For `root@yagoseek.dev`, the authorized production exception creates no new pre-upgrade backup. Preserve every existing archive under `/opt/yago/backups` unchanged, stage and verify the published package under `/opt/yago/releases/v0.0.14`, stop both services, install it, then manually start and verify the node before the crawler. Do not open, replace, prune, or recreate the live persistent databases.

The durable formats are unchanged. A package rollback to matched v0.0.13 binaries is mechanically possible after stopping crawler before node and starting node before crawler, but it restores the production-acceptance defect corrected here. Under the affected load, keep the v0.0.13 crawler stopped. Restore a matching stopped backup only when a point-in-time rollback is required; doing so discards progress committed after that backup.

## Known limitations

- The active-task limit is per crawler process, not fleet-wide. Aggregate active tasks equal the sum of connected process limits, and there is still no global pages-per-second ceiling.
- Admission is completion-driven. Lowering the limit does not preempt long-running work, and a task can hold one slot until it reaches terminal completion or is cancelled through the existing control path.
- Running progress coalescing deliberately omits superseded intermediate snapshots. Terminal state, phase order, and the latest pending running state remain protected.
- bbolt still permits one read-write transaction at a time. Shorter ingest authorization and bounded active work reduce control-plane convoys; they do not make durable writes parallel or remove storage latency.
- The crawler remains at-least-once. A page whose response began but whose durable outcome did not commit before termination may be fetched again.
- Recovery remains tied to the configured node and crawler data directories. This release adds no cross-host frontier replication, distributed lease catalog, or fleet-wide checkpoint protocol.
- This maintenance release does not change search recall, morphology, ranking, PDF extraction, browser rendering, Tavily compatibility, or public portal behavior.
