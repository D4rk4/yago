## Abstract

YaGo v0.0.9 corrects cold-start `Always` web-search fusion, adds bounded automatic-discovery priority and live per-crawler fetch concurrency, and revises the Admin console with paged activity, a Photon-style shelf and controls, cache-safe embedded assets, and a status-bearing login. Swarm-derived crawling is enabled by default at depth 5 and 250 pages per host; web-discovery crawling remains disabled by default. The release adds crawler/browser and crawl-index write metrics, exact release and dated-development build identities, deterministic module hygiene, and attested release-asset provenance. YaCy search and protocol surfaces, listeners, and canonical crawl-order storage remain compatible. Operators should review crawler defaults and back up `/opt/yago/data` before upgrading.

## Changes

### Search execution and ranking continuity

- `Web search fallback (DDGS) = Always` starts eligible web retrieval concurrently with local and YaCy-swarm retrieval even on a cold process and even when a local or peer hit is eventually available.
- The concurrent primary exact stage receives 1,400 milliseconds in `Always` mode instead of retaining the 600-millisecond sequential-fallback reservation. The bounded 150-millisecond local recovery stage and 1.8-second hard search boundary are unchanged.
- A full-assembly regression covers a slow first local hit and a web hit in the same response, preventing startup timing from turning `Always` into miss-only fallback behavior.
- Learned local candidates retain the bounded relevance of the local fusion position they occupy before diversity selection. Raw model values no longer compete directly with peer or web lexical scores.
- The mixed-source ranking path adds no request-time network, storage, or model-loading operation and does not expand the candidate window.

### Crawler scheduling and live capacity

- Automatic swarm discovery is enabled by default with crawl depth 5 and 250 pages per host. Web-discovery crawling remains disabled by default but is preconfigured with the same depth and page cap.
- Both automatic profiles use the HTTP fast path, allow query-bearing URLs and untrusted certificate authorities, honor `robots.txt`, skip `rel=nofollow` links, and use a 30-day recrawl interval.
- Automatic-discovery work carries explicit priority metadata; profile names are not interpreted as policy. The default scheduler serves no more than three waiting discovery orders before a normal order and no more than three due discovery pages before a normal page, so priority remains work-conserving and bounded.
- Disabling automatic-discovery priority restores global FIFO order leasing and the existing run-fair, value-scored page selection without class preference.
- Priority survives requeue, lease expiry, and restart. Canonical order payloads remain in the existing `crawlorders` bucket; key-only secondary indexes and a durable watermark reconcile work admitted or consumed by an older node.
- `Maximum fetch concurrency per crawler` is a live setting with a range of 1–256 and a default of 4. Each connected crawler drains its active fetch group and applies the latest value without a process restart.
- `YAGOCRAWLER_WORKERS` is now the matching node and crawler bootstrap value. `YAGO_CRAWLER_PRIORITIZE_AUTOMATIC_DISCOVERY` is the matching bootstrap for both processes. A bounded startup heartbeat applies the node's persisted policy before intake when available; later heartbeats converge a temporarily disconnected crawler.
- The limit is per crawler process, not a fleet-wide cap. Aggregate capacity is the configured value multiplied by the number of current crawler processes.

### Admin console

- Search Activity renders the newest 20 retained requests per page and uses the same server-rendered Previous/Next navigation as the Crawler and Network tables. Lifetime totals and full-ring top-word statistics remain independent of the visible page.
- Configuration → Crawler now owns separate `Crawler`, `Automatic discovery`, and `Document formats` fieldsets. The former standalone Autocrawler route redirects to the canonical tab instead of maintaining a second settings surface.
- Crawler controls remain on one line, and the pages-per-minute input displays the effective value even when it comes from the default.
- The Photon theme now uses a compact contiguous tab strip, raised and pressed push-button states, continuous field-row separators, structural property and multi-column table treatments, and a flat right-side shelf.
- The shelf uses 15 locally embedded color SVG assets from the MIT-licensed Haiku icon set. It retains one visible, keyboard-accessible route list without collapsible groups.
- System Monitor shows sampled process CPU, process resident memory, host used/total/available memory, and data-vault use against quota as bounded meters. When current crawlers are connected and report the optional activity measurement, it also shows busy fetch jobs against aggregate live capacity; the crawler row is hidden when no crawler is connected and reports unavailable for a mixed legacy fleet.
- The unauthenticated login follows the requested QNX Neutrino composition while retaining CSRF, rate limiting, strict CSP, and anti-enumeration behavior. It leaves the account name empty and presents independently bounded node name, swarm address, processor model, logical CPU count, memory, free data storage, build identity, and uptime facts.
- Logout invalidates the session and returns to a clean login form without a redundant signed-out banner.
- Embedded Admin assets are bound to their exact content digest. Only the canonical path with the current single `v` value is immutable; unversioned compatibility paths revalidate, while stale, foreign, duplicated, malformed, or aliased revisions return a no-store 404.
- The pre-auth stylesheet uses digest-bound canonical-path validation and filesystem-bounded serving. Tests cover stale, duplicated, encoded, malformed, and aliased revision requests without scanner suppression.

### Observability

- The node exports `crawl_search_index_write_duration_seconds` for every logical local-index write attempt, `crawl_search_index_documents_total` only after a successful group, and `crawl_search_index_write_failures_total` on failed groups.
- The crawler exports `yacy_crawler_browser_slot_acquisition_deadlines_total`. It advances only when acquisition of a Firefox selection or session slot ends in `context.DeadlineExceeded`; ordinary cancellation and crawler shutdown are not counted.
- All new series are fixed-cardinality and unlabeled. They add no search-path I/O.
- Storage-read failures publish an unavailable value rather than an invented zero, and byte-valued Performance history uses binary human-readable units.

### Build and release discipline

- Release binaries retain the exact `vN.N.N` identity. Development container builds receive an explicit `YYYY.MM.DD-dev` identity from the invoking Make target, so the date participates in Docker's build cache key; a missing or malformed identity is rejected.
- `make verify` now begins with `go mod tidy -diff` for every module and rejects an uncommitted module-graph rewrite before formatting, analysis, tests, coverage, or builds.
- The release workflow requires the committed memo for its tag, validates the bounded Abstract and stable section order, and publishes that text instead of an autogenerated commit list.
- Each tarball, Debian package, and RPM receives GitHub-hosted build provenance after package construction and the applicable amd64 smoke tests. The release job verifies every downloaded attestation against the release workflow, tag ref, and source commit before publishing the assets.
- Provenance binds an artifact digest to this repository's workflow and source revision. It does not claim that the artifact is safe or reproducible.

## Compatibility

- YaCy `/yacy/*`, public search, OpenSearch, and Tavily-compatible response surfaces are unchanged.
- No port, listener, service, or required runtime dependency is added.
- Existing crawl-order JSON remains readable. The new optional priority field is ignored by older readers, and absent or unknown priority is normal work.
- The crawler heartbeat adds optional `active_fetches`. An explicit zero means measured idle; omission identifies an older crawler. Older protobuf readers ignore the field.
- Two additive crawl-control kinds carry fetch concurrency and automatic-discovery priority. Older crawlers ignore unknown kinds and retain their bootstrap values; current crawlers connected to an older node do the same.
- A v0.0.8 node can drain every pending order in FIFO order because payloads remain in the established bucket. Returning to v0.0.9 reconciles additive secondary indexes and removes stale keys lazily.
- The legacy `/admin/autocrawler` GET and POST locations redirect to Configuration → Crawler.
- Package metadata remains numeric `0.0.9`; `--version` output includes the leading `v` as `yago-node v0.0.9` and `yago-crawler v0.0.9`.
- Direct Dockerfile or Compose builds must supply a valid `VERSION`; the project Make image targets supply the UTC development identity automatically.

## Upgrade and rollback

Review the automatic-discovery defaults and keep the node and crawler bootstrap values for `YAGOCRAWLER_WORKERS` and `YAGO_CRAWLER_PRIORITIZE_AUTOMATIC_DISCOVERY` equal. Runtime values saved through Admin become authoritative after connection.

1. Download the architecture-matching package and verify its GitHub attestation:

   ```sh
   gh release download v0.0.9 --repo D4rk4/yago \
     --pattern 'yago_0.0.9_amd64.deb' --dir /tmp/yago-v0.0.9
   source_digest=$(gh api repos/D4rk4/yago/commits/v0.0.9 --jq .sha)
   gh attestation verify /tmp/yago-v0.0.9/yago_0.0.9_amd64.deb \
     --repo D4rk4/yago \
     --signer-workflow D4rk4/yago/.github/workflows/release.yml \
     --source-ref refs/tags/v0.0.9 \
     --source-digest "$source_digest"
   ```

2. Take a consistent backup of `/opt/yago/data` on a filesystem with sufficient free space.
3. Stop both services:

   ```sh
   sudo systemctl stop yagocrawler yago-node
   ```

4. Install the published package:

   ```sh
   sudo dpkg -i /tmp/yago-v0.0.9/yago_0.0.9_amd64.deb
   ```

5. Start the node, verify health and readiness, then start the crawler:

   ```sh
   sudo systemctl start yago-node
   wget -qO- http://127.0.0.1:9090/health
   wget -qO- http://127.0.0.1:9090/ready
   sudo systemctl start yagocrawler
   systemctl --no-pager --full status yago-node yagocrawler
   ```

The package enables but does not start the units. For rollback, stop both services, install the v0.0.8 package, restore the stopped pre-upgrade backup, start the node and verify readiness, then start the crawler. A v0.0.8 node can read and drain pending order payloads but ignores priority indexes, live concurrency controls, and the additive heartbeat measurement.

## Known limitations

- `Always` remains an explicit operator setting. External web-provider availability is not guaranteed, and only results completing inside the fixed search budget can contribute.
- Web-discovery crawling remains disabled by default. Enabling it can admit provider-discovered URLs into the crawler subject to normal safety and capacity policy.
- A mixed fleet containing crawlers older than v0.0.9 cannot report a truthful aggregate busy-worker numerator; System Monitor displays that value as unavailable until all connected crawlers support it.
- Fetch concurrency is a per-process setting, not a global fleet admission limit.
- System Monitor is sampled on a bounded ten-second interval. Its first process-rate reading is unavailable until the next sample, and it does not persist history across restart.
- Browser slot deadlines identify saturation but do not diagnose the remote page or Firefox failure that consumed the slot.
- Sitemap `lastmod` scheduling, browser-level redirect interception, hardened bot-wall handling, and the remaining persistent-frontier refinements are not completed by this release.
- Release container images are not published; build them from the pinned project Dockerfiles through the supplied Make targets when required.
