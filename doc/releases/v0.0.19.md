## Abstract

YaGo v0.0.19 repairs Tavily-compatible API authentication for Admin-minted scoped credentials. The public agent surface now shares the operations credential authority in both compatibility and scoped-only modes; the switch disables only the optional legacy static token. Persisted credential corruption is reported as 503 rather than an invalid-key 401, and Admin text now states the effective scope requirements. No listener, port, storage schema, YaCy wire, crawler, search-ranking, or response-shape change is introduced. Existing scoped keys remain valid after upgrade and vault reopen. Rollback to v0.0.18 restores the former rejection behavior when scoped-only enforcement is off.

## Changes

### Scoped credential authority

- The node now connects the Admin API-key authority to the Tavily-compatible public surface during every boot. Admin-minted keys no longer depend on the scoped-only switch being enabled.
- With scoped-only enforcement disabled, either an Admin-minted key holding the required scope or the exact configured legacy static token can authorize a request. With enforcement enabled, only scoped credentials are accepted.
- The legacy static token is compared before scoped lookup. A nonmatching bearer value continues through the scoped store so valid Admin-minted keys and the store's failure distinctions are preserved.
- Ordinary `POST /search` requires `search:read`. Raw-content search, `POST /extract`, `POST /crawl`, and `POST /map` require `search:raw`.
- The operations and public listeners share one in-process credential authority. A key minted through the guarded operations API is immediately valid on the public listener and remains valid after the durable vault is closed and reopened.

### Authentication status and storage integrity

- Missing, malformed, invalid, or revoked credentials return 401 with a Bearer challenge. A valid key without the required scope returns 403. Per-key throttling returns 429 with retry guidance. Credential-store admission, lookup, or record-decoding failure returns 503 with retry guidance.
- Stored credential records must contain the canonical lowercase hexadecimal encoding of one SHA-256 value. A malformed or noncanonical record is classified as unavailable credential storage instead of being indistinguishable from an invalid caller credential.
- Legitimate keys created by YaGo already use this canonical representation; the validation adds no storage migration and does not rewrite existing records.
- Test failure paths no longer print generated API keys, login response bodies, CSRF tokens, or recoverable credential material.

### Operator surfaces and documentation

- Admin Configuration names the switch **Require scoped API keys only** and identifies `YAGO_SEARCH_API_KEY` as the legacy static credential.
- Admin Security lists all four Tavily-compatible routes and states the `search:read` and `search:raw` requirements without implying that scoped keys work only when enforcement is enabled.
- The specification, compatibility catalog, configuration parity record, README, and feature catalog now describe the same dual-mode policy and the 401, 403, 429, and 503 outcomes.

## Compatibility

- `YAGO_SEARCH_REQUIRE_API_KEY=true` retains its secure operational result: the legacy static token is rejected and scoped keys are required. The corrected behavior is visible when the value is false: valid scoped keys are now accepted alongside the optional static token.
- `YAGO_SEARCH_API_KEY` keeps its name, default, and bootstrap role. An empty value does not make the agent API public; a caller must still present a valid scoped credential.
- No environment variable, Admin setting key, route, request field, response field, listener address, port, service, package path, or container name changes.
- No YaCy peer protocol, RWI, DHT, crawler frontier, document-store schema, Bleve mapping, query planning, ranking, or web-fallback behavior changes.
- Existing YaGo-generated key records remain readable. A manually altered or corrupted noncanonical secret hash now returns 503 until the operator revokes or repairs that record instead of returning 401.
- Upgrade both packaged binaries together. The crawler binary is unchanged in behavior, but it remains part of the single coordinated YaGo package and release identity.

## Upgrade and rollback

1. Download the architecture-matching published v0.0.19 package and verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.19`, and the tagged source commit.
2. Follow the target's documented data-protection policy. Generic bare-metal nodes take a coordinated stopped backup; the documented `root@yagoseek.dev` exception preserves existing archives and stages the package under `/opt/yago/releases/v0.0.19` without creating a new backup.
3. Stop `yago-crawler` and `yago-node`, install the package with `dpkg -i`, and confirm that package installation did not start either service automatically.
4. Start `yago-node` first. Verify the package and binary version, `/health`, `/ready`, the public and operations listeners, and one disposable scoped-key request through the external proxy.
5. Start `yago-crawler` only after node readiness. Verify its exact version, active systemd state, node connection, and advancing heartbeat.

For rollback, stop the crawler and node, reinstall the v0.0.18 package, start and verify the node, then start the crawler. This release introduces no data-format migration, so a data restore is not required solely because of this change; retain and use the target's coordinated backup when rollback follows unrelated data damage. After rollback, scoped keys are again rejected while scoped-only enforcement is disabled, which is the v0.0.18 defect corrected here.

## Known limitations

- This release cannot establish whether a credential copied from another application's environment was minted by the serving node. Invalid, revoked, foreign-node, or mistyped credentials correctly remain 401. Create the integration credential on the node that serves the request.
- Scoped credentials are node-local. This release does not replicate the Admin credential store across YaGo peers or introduce a cross-node credential authority.
- The legacy static token has no individual scope set, revocation record, per-key audit identity, or scoped-store rate history. It authorizes every Tavily-compatible capability while configured and allowed. Operators that do not require compatibility should enable scoped-only enforcement and use Admin-minted keys.
- Credential-store unavailability is reported as 503 with one-second retry guidance; the node does not repair corrupted records or retry a request after responding. Operators must restore store availability or revoke the affected record.
- Last-used telemetry remains best effort and coalesced. Successful authorization is not failed merely because a last-used update could not be persisted.
- This release changes authentication selection only. It does not add semantic reranking, model-guided crawl, an upstream Tavily provider, credential federation, or new Tavily request and response fields.
