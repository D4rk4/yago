## Abstract

YaGo v0.0.15 closes crawler fleet-control, storage-contention, and stock-YaCy interoperability gaps. The node now owns one non-bursting fetch-start ceiling, distributes authoritative crawler policy and URL denylist revisions, repairs oversized automatic-discovery checkpoints, and supports bounded authenticated remote crawl on controlled networks. Crawler operations gain saved profiles, per-URL outcomes, hardened Firefox launch and redirect handling, and complete configuration parity. Storage moves collection-length accounting off a shared writer hotspot, bounds ingest and wire transfers, reclaims posting-only lineages, and emits Java-consumable remote-search evidence. Operators must upgrade node and crawler together; finite-rate mixed versions fail closed, and rollback after new vault mutations requires a coordinated backup.

## Changes

### Fleet-wide crawler admission

- `crawler.max_pages_per_second` is now one node-owned ceiling across every connected crawler process and active run. The default remains 10 fetch starts per second; zero remains unlimited.
- The node issues short, fair, non-bursting permit windows. Missed permits are not reclaimed, finite schedules start with a quiet interval after restart, and rate reductions fence permits issued under the earlier generation.
- Crawlers retain a local process smoother and request only demand-backed permit batches bounded by live fetch-worker concurrency. Per-run pace, per-host delay, and worker concurrency remain additional limits.
- A permit whose usable window is consumed by round-trip time is discarded and retried without a catch-up burst. Sequence replay is idempotent after a lost response.
- A finite ceiling requires fetch-start lease support on both sides. An incapable node or crawler cannot silently bypass the configured limit.

### Authoritative crawler runtime policy

- Configuration → Crawler is authoritative for browser, network, redirect, depth, host concurrency, crawl delay, timeout, sitemap, shutdown, rate, worker, and process-facility policy. Environment variables are bootstrap defaults.
- A crawler reads the typed policy before assembling its fetch stack and receives later changes through heartbeat. Changes that cannot be applied in place request a graceful crawler restart.
- Browser-sandbox changes let an active render finish and retire the affected pooled Firefox session before its next render.
- A configured or discovered Firefox launcher must resolve through root-owned, non-group-writable and non-world-writable path components to a regular non-set-ID `firefox` or `firefox-esr` executable available to the crawler identity. The crawler checks the boundary before assembly and immediately before every spawn.
- The optional crawler metrics listener accepts only an empty value or a loopback IP-literal address. Remote scraping requires a trusted local tunnel or proxy.
- Private-network admission is limited to explicit RFC 1918 or IPv6 ULA policy. Loopback, link-local, metadata, carrier-grade NAT, multicast, unspecified, and reserved destinations remain denied.

### Frontier budgets, denylist, and crawl evidence

- The visible 250-page swarm and web-discovery values now limit the complete automatic task as well as each host. A lower positive global run cap may reduce the task but cannot enlarge it. Web discovery remains disabled by default.
- Recovery repairs legacy automatic checkpoints whose missing, unlimited, or excessive whole-run budget would exceed the stored per-host ceiling. It removes the newest pending pages in bounded atomic batches while preserving completed totals, visited history, host admission state, and the oldest pending work.
- Durable discarded-page accounting makes repeated crash recovery idempotent. A run whose completed work already reaches its cap removes the remaining pending pages and settles normally.
- The Index URL/domain denylist is delivered to crawlers as a bounded, revisioned snapshot. Initial delivery fails closed; an invalid later update cannot replace the last valid policy.
- Denylist policy is enforced before seed and discovered-link admission, around HTTP, sitemap, and browser fetches, and against final redirected URLs.
- Named crawl profiles can be created, updated, applied, and removed from Admin. Saved profiles exclude seeds and affect only future dispatches.
- Run detail records retain at most 64 bounded per-URL outcomes with one terminal class, a stable reason, and either the known HTTP status or `Unavailable`. Whole-run and per-host maxima are reported independently.
- Aggregate failure rate uses `failed / (fetched + failed)` and remains bounded to 100 percent even when an accepted response later fails parsing or indexing. Parser failures also have a separate crawler metric.

### Controlled-network YaCy remote crawl

- Controlled private networks may select Java-compatible `salted-magic-sim` signing and validation with one nonempty shared secret. Unknown authentication modes fail closed.
- Remote crawl remains disabled by default and is advertised only when authentication, the shared secret, trusted peer hashes, and positive destination policy are all complete.
- An enabled node copies eligible local URL orders asynchronously into a separate durable delegation queue without delaying or removing the authoritative local order.
- Exact authenticated peers may lease bounded single-URL work. Request rate, outstanding leases, lease expiry, queue capacity, pending order, and requeue state survive restart.
- Destination domain or IP-prefix policy and DNS answers are checked at staging, lease, and receipt time. Only a matching, unexpired `fill` receipt may store bounded URL metadata and complete the lease.
- Remote delegation transfers URLs and bounded metadata only. It does not transfer bodies, crawler profiles, redirects, or follow-up depth.

### YaCy interoperability and storage throughput

- Inbound RWI and URL transfers accept at most 1,000 rows. An oversized RWI tail is no longer silently truncated and acknowledged.
- Saturated `transferRWI` admission returns YaCy's HTTP 200 `too high load` response. Oversized transfers and pre-commit storage or deadline pressure return HTTP 200 `busy` with a millisecond pause; `transferURL` uses its parseable not-granted response.
- Grouped crawler ingest sends postings in retryable chunks of at most 8,192 instead of allowing one sixteen-delivery group to become a 131,072-row storage transaction.
- Exact collection-length additions and removals are recorded on each record's physical shard rather than one fixed counter shard. Existing totals remain the upgrade base, and old vaults open without a scan.
- DHT transfer tallies are coalesced for one second and flushed as independent small transactions. Live reports include pending values, and graceful shutdown attempts to drain them.
- Inbound URL reconciliation uses a bounded in-memory identity observation set instead of synchronous reference reads on each transfer.
- Remote search attaches a transient complete enhanced-base64 `wi` WordReferenceRow to each response copy. Stock Java YaCy accepts that ranking evidence; stored URL metadata remains unchanged.
- Accepted remote-search resources contribute to both advertised received-word and received-URL totals.
- When ordinary stale-URL eviction cannot reach the soft quota target, a bounded posting scan can reclaim URL hashes that have RWI postings but no URL-metadata row.

### Admin and operator surfaces

- Admin Search now carries content-domain, language, and site-host filters through the shared search request and pagination path. Each query links to the matching local or global Search Explain view.
- Public portal and `/yacysearch.html` rows carry up to six bounded human-readable ranking reasons derived from evidence already computed by the request. They do not start another retrieval or alter ordering.
- Index adds restart-safe full-index rebuild scheduling and bounded escaped document inspection for stored metadata, link evidence, images, and extracted text.
- Network adds an explicit CSRF-protected public-endpoint self-test that performs one fresh bounded observation and records the operator outcome.
- Logs add composable inclusive UTC `from` and `to` filters. Invalid or reversed ranges produce a stable validation response instead of misleading rows.
- Listener controls persist a disabled public or crawler listener, expose Reset only for stored overrides, and restore environment authority when reset.
- Persisted outbound egress policy is now applied before the shared outbound client is constructed, closing the restart path where an Admin override could be displayed but not enforced.
- Active Admin sessions rotate to a fresh unpredictable cookie token after the earlier of one hour or half their lifetime while preserving CSRF identity and absolute expiry.
- Node and crawler storage-reserve facts are shown on Index rather than in the shelf System Monitor. The crawler fetch meter remains conditional on a connected reporting crawler.
- The Neutrino-inspired tables, tabs, controls, and narrow layouts were rechecked after the operational changes; no new shelf item or monitor row was introduced.

### Configuration and deployment parity

- An AST-backed catalog classifies every canonical node and crawler environment input and rejects future unclassified environment-looking constants and lookups.
- Each mutable behavior control maps to a persisted Admin setting or a bounded dedicated Admin surface. Shared validators keep bootstrap and Admin defaults and limits aligned.
- `.env.example`, the canonical Compose example, and both systemd environment examples now cover the same canonical controls while excluding migration-only aliases.
- The crawler execution depth ceiling defaults to 5, matching the shipped automatic-discovery profiles.
- Logging level is a persisted live setting. Listener, network identity, authentication, DHT geometry, browser path, metrics address, storage, egress, web-fallback, and extract-fetch values use shared bounded validation.

## Compatibility

- Default public `freeworld` behavior remains uncontrolled and interoperable. Controlled-network authentication and remote crawl are explicit opt-ins.
- The YaCy crawler-control changes do not add YaCy wire fields. Fetch-start leases, runtime policy, URL outcomes, immutable run limits, denylist revisions, and active-fetch observations are additive node-crawler protobuf surfaces.
- Older protobuf decoders ignore the additions, but the default finite fetch-start ceiling deliberately makes mixed-version node/crawler operation unsupported. Upgrade both binaries together.
- Existing crawler checkpoints and broker state open in place. Legacy automatic tasks may have excess pending discovery pages removed to enforce their stored visible cap; completed and visited facts are retained.
- Existing vaults upgrade to shard-local collection-length deltas without a scan or reindex. Once current binaries record new mutations, an older binary cannot reconstruct exact collection lengths from the legacy counter.
- No automatic Bleve rebuild is performed. The Admin rebuild action only schedules bounded startup work for the next node restart.
- Public result reasons and Admin filters consume existing search evidence and request fields. They do not change YagoRank coefficients, learned models, Tavily response ordering, or YaCy RWI addressing.
- The RWI responder remains a compatibility and exchange path backed by RWI postings and URL metadata, not Solr.

## Upgrade and rollback

1. Download the architecture-matching published v0.0.15 package and verify its GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.15`, and the tagged source commit.
2. Before stopping services, verify that node and crawler bootstrap values agree. An explicit Firefox path must satisfy the new trust checks, and `YAGO_CRAWLER_METRICS_ADDR` must be empty or loopback-only. Leave remote crawl disabled unless every controlled-network prerequisite is intentional.
3. On a generic bare-metal installation, back up the complete node and crawler data directories.
4. Stop `yago-crawler`, then stop `yago-node`. Install the matched package containing both binaries with `dpkg -i`; package installation enables but does not start them.
5. Start `yago-node` first and verify `/health` and `/ready`. Start `yago-crawler` only after node readiness.
6. Verify one stable crawler session, advancing fetch-start leases, progress without repeated control-RPC timeouts, automatic-discovery caps, Index storage status, and the intended public and crawler listener states.

For `root@yagoseek.dev`, the authorized production exception creates no new pre-upgrade backup. Preserve every existing archive under `/opt/yago/backups`, stage and verify the package under `/opt/yago/releases/v0.0.15`, stop both services, install it, then manually start and verify the node before the crawler.

Rollback is coordinated and backup-based. Stop crawler before node, restore a matching stopped node data directory, broker database, crawler checkpoint, and search state, install matched older binaries, then start node before crawler. Do not open data mutated by v0.0.15 with v0.0.14: shard-local length changes are not reconstructible by the older binary. On the production exception target, rollback is unavailable unless an existing compatible coordinated backup is retained.

## Known limitations

- bbolt still permits one read-write transaction per database. Shard-local accounting, bounded transactions, and coalesced tallies reduce contention; they do not eliminate storage latency.
- A crash can lose in-memory transfer-tally observations since the last successful flush when persistence failures span several cycles. Stored RWI and URL data remain authoritative.
- The crawler remains at-least-once. An in-flight page whose durable outcome did not commit may be fetched again.
- Frontier checkpoints remain local to each crawler `YAGO_DATA_DIR`. Moving an unexpired lease to another host requires moving or sharing that directory; there is no cross-host frontier replication.
- Active-task admission remains per crawler process and does not preempt long-running work. The fetch-start rate is fleet-wide, but there is no fleet-wide active-task ceiling.
- High control-plane round-trip time can consume a short fetch-start permit window. The crawler discards that permit rather than violating the ceiling, so achieved throughput may be below the configured rate.
- Remote crawl remains partial and default-deny. It can duplicate the authoritative local fetch and does not delegate bodies, profiles, redirects, or follow-up depth.
- The node-crawler gRPC transport remains plaintext. A non-loopback crawler listener belongs only on a trusted private network or authenticated tunnel; `salted-magic-sim` protects YaCy request identity and is not transport encryption.
- Browser executable checks cover POSIX ownership, mode bits, executable identity, and set-ID state. ACLs, mount policy, and mutation by a trusted root operator remain outside that boundary.
- Browser fallback remains a bounded Firefox-based heuristic. It performs no OCR, is not a general browser automation service, and keeps public-web admission, proxy DNS checks, redirect limits, and timeouts as defense in depth.
- Stock Java can derive a different local-domain URL-hash flag for an unresolvable hostname under an unrecognized top-level domain. Exact cross-peer URL identity is claimed for recognized public-domain inputs, not that synthetic edge.
- Remote RWI search remains a bounded conjunctive compatibility path and ignores Solr-only ranking fields. Local Bleve search remains authoritative for full-text retrieval and snippets.
- Public ranking reasons are bounded summaries of already computed evidence, not a complete causal proof or a second Search Explain execution.
- Full-index rebuild is scheduled for restart and remains subject to ordinary storage-growth admission and available disk headroom.
