## Abstract

YaGo v0.0.20 corrects operational failures across Tavily extraction, crawler warming and restart recovery, document admission, anchor projection, storage capacity observation, and peer reachability. Tavily extraction no longer lets a contended stored-document lookup consume the request budget, and operator-enabled web results enter bounded crawler warming. Exact URL arbitration and transaction-bounded anchor publication remove unrelated contention and oversized retry loops. Validated external hello back-pings now take precedence over NAT-sensitive self-probes; junior callers remain visible but non-routing, and listeners bind before announcements. A crawler abandons dead stream attempts and resumes durable leases after either service restarts independently. No wire field, stored format, route, setting key or default, listener address, topology, or dependency changes. Upgrade node and crawler together; retained state requires no migration.

## Changes

### Peer-confirmed public reachability

- Outbound `/yacy/hello.html` processing now validates the reported caller address and peer type, requires a responder seed, and requires that seed's hash to identify the peer that was contacted. A malformed or substituted response cannot become reachability evidence.
- Only a peer whose primary advertised address is a public IP literal can act as an external observer. A successful validated greeting records its `senior`, `principal`, or `junior` classification for 15 minutes. Evidence is keyed by at most 1,024 observer hashes. When the ordinary announcement interval is slower, an additional greet cycle runs every ten minutes to provide bounded refresh opportunities.
- At least one current `senior` or `principal` observation reports the advertised endpoint reachable. A `junior` report replaces only that observer's prior positive report; one observer cannot erase another observer's current success. An all-`junior` current set reports unreachable.
- Any current peer observation is authoritative. The bounded direct RWI-capacity query is considered only when no current peer observation exists, and executes only when `YAGO_PUBLIC_SELF_TEST_URL` explicitly pins a public base URL. An automatically derived local or loopback target remains unconfirmed and is not accepted as proof of public ingress.
- A greet transport failure records no classification because inability to reach the observer says nothing about inbound reachability to this node.
- Admin Network now renders `Unconfirmed`, `Reachable`, or `Unreachable`, together with peer-back-ping or configured-direct-query provenance and a UTC observation time when known. Its CSRF-protected action evaluates the same evidence boundary and records the outcome as an operator event.
- Admin Overview no longer copies the compatibility seed's advertised senior type. It reports `senior` for current positive external evidence, `junior` for a current all-junior set, and `unknown` when no current external classification exists.

### Listener readiness before peer presence

- The peer, operations, and optional public-search HTTP listeners are all bound before peer greeting and announcement loops start. The first outgoing hello can no longer race the advertised peer listener into a false junior result.
- If any configured HTTP bind fails, startup closes every HTTP listener already opened and starts no peer-presence work. Listener addresses, ports, handlers, and shutdown ordering are otherwise unchanged.

### Bounded junior potential peers

- After network authentication, an inbound hello caller is observable only when its seed has a usable endpoint or its advertised port can be combined with the trusted transport address. An unspecified advertised IPv4, IPv6, or mapped-IPv4 literal uses that transport fallback rather than entering the roster. Self identity, a missing port, an unspecified or invalid transport address, and an otherwise unusable seed are not retained.
- Callback work has one aggregate 6.5-second HTTP budget or 13-second HTTPS-first budget, matching the upstream YaCy hello boundary and leaving headroom inside the 30-second inbound request. A valid nonnegative RWI-count response is required; an HTTP 200 response containing a negative result is unreachable.
- The local callback result replaces the caller's claimed type. A successful callback retains or promotes the caller as senior; a failed callback retains or demotes it as junior. The resulting caller observation has a separate one-second persistence context, so a timed-out or disconnected hello request cannot erase a legitimate junior observation. The observed seed and local type use the existing persistent peer encoding and the existing 4,096-peer reservoir bound.
- Locally junior callers remain visible in the complete Admin Network roster and known-peer total. They do not enter the 256-peer active set, reachable totals, hello seed replies, seed-list export, remote-search selection, or DHT target selection. Blocking remains an independent exclusion, and the Admin reachable total is derived after the block filter.
- Caller news is accepted only after the same authenticated observable admission. Failed persistence does not mutate active membership.

### Independent crawler and node restart recovery

- Each crawler order-stream attempt now carries its own cancellation context through stream receipt, lease confirmation, and delivery to local active-run admission.
- When a node restart invalidates the process session while a confirmed order is waiting behind local admission, cancellation releases that obsolete delivery. The running crawler opens a new stream, adopts its durable leases, resumes unfinished work, and then accepts new work instead of repeating `crawl lease lost` indefinitely.
- An independently restarted crawler retains its checkpoint-backed worker identity, creates a replacement process session, and adopts the same worker's unfinished leases while the node remains running.
- The crawler RPC shape, lease encoding, worker checkpoint, frontier database, and operator settings are unchanged. At-least-once semantics remain: an in-flight page whose outcome was not committed can be repeated after recovery.

### Exact and fair document URL admission

- Stored-document exclusion now uses an exact URL registry rather than 4,096 hash stripes. Unrelated URLs no longer serialize because two hashes selected the same stripe.
- Each URL has an ordered reader/writer queue. Consecutive readers at the head can proceed together; a later reader or writer cannot cross an earlier writer. Multi-URL callers remove empty and repeated identities and acquire the remainder in sorted order.
- Cancellation removes a queued waiter or releases an admission won concurrently with cancellation. Stored-document and presence reads therefore return through their request context rather than remaining parked behind a writer.
- The registry retains at most 4,096 idle boundaries in oldest-idle order. Active and referenced boundaries remain until their final release, so the idle limit never evicts a lock in use.
- Document and anchor paths measure storage capacity before entering their write and exact-URL boundaries; a shard-wide capacity observation does not hold those exclusions while it waits.
- Canonical ingest and stored-cluster revision snapshots retain caller-submitted inbound evidence separately from anchors materialized by the anchor row. Final `Receive` processing restores only the submitted evidence, then rereads and canonicalizes the current anchor row under the exact-URL writer; an anchor replacement between snapshot preparation and commit therefore cannot preserve both old and current same-source revisions. The ordinary read-only document path remains unchanged.

### Transaction-bounded outbound-anchor replacement

- The existing crawl-ingest group remains limited to 64 deliveries and 64 MiB of encoded JSON. Its anchor subphase now divides the group into ordered source replacements of at most 16, matching the document-store replacement limit.
- Each source group computes the sorted union of prior and desired targets. It locks and projects at most 16 exact target URLs at a time. The main document-write admission is released before the index callback, while the current target-page locks remain held through that callback; later pages and unrelated URLs remain available.
- One target mutation transaction contains at most 32 physical rows and 8 MiB of encoded keys and values. Byte pressure can split one 16-target page into more than one transaction. The at-most-16 sorted source publication rows are partitioned into deterministic subtransactions whose conservative encoded ceiling is at most 64 MiB; one source row above that ceiling is rejected before target projection.
- All changed-source contributions in one source group are aggregated and projected once before any final publication subtransaction runs. Capacity, page storage, or index callback failure therefore leaves every prior source publication authoritative.
- A later publication-subtransaction failure can leave earlier source rows at the desired revision and later rows at their prior revision. The lineage reservation remains held across every subtransaction; redelivery recomputes the remaining aggregate, treats completed target and publication work idempotently, and converges before URL metadata, RWI postings, acknowledgements, or later tail work advances. Document-lineage clearing uses the same paged path.

### Capacity observation under continuous mutation

- Concurrent capacity callers share one measurement. A successful live-byte observation remains reusable for at most one second even while storage mutations continue, avoiding a complete vault measurement for every admission during sustained ingest.
- If the measurement leader is cancelled or reaches its deadline, a healthy waiter may open a replacement measurement. A non-context storage error is returned to the leader and every caller already waiting on that measurement instead of being erased into an immediate internal retry.
- Failed observations are not cached. The next independent caller wave can retry, while one persistent failure cannot spin inside the same request.

### Bounded web-result warming

- Operator-enabled web-discovery warming now uses two process-wide workers and at most 128 pending jobs. Search responses do not wait for crawler publication.
- Each job receives a fresh ten-second background context when execution starts, so queue delay does not consume its publication budget or retain request cancellation and values after the response.
- URLs are parsed before admission; fragments are removed, pending normalized identities coalesce, and credentials, missing hosts, non-HTTP schemes, and identities above 2,048 bytes are rejected.
- Each eligible URL receives a stored-document lookup bounded to 50 milliseconds. An absent or lookup-indeterminate URL attempts bounded `PublishOnce` delivery under the existing automatic-discovery profile. At most one accepted durable order remains for that normalized URL.
- The discovery root is the warming fetch. The former parallel one-page cache order is removed, so one surfaced row cannot schedule two independent root fetches.
- A full pending queue warns and drops only the new optional warming job. Bounded publication retries, crawler availability, robots policy, egress policy, page budgets, and durable frontier deduplication remain authoritative.

### Bounded ordered extraction

- Tavily-compatible `POST /extract` now gives each stored-document lookup at most 250 milliseconds. A stored hit remains preferred and starts no outbound request.
- When lookup reaches that boundary and guarded fetch-on-extract is enabled, resolution continues with the request's remaining budget. The existing egress guard, per-fetch timeout, response-size ceiling, and raw-content hard deadline remain in force.
- A lookup timeout without an enabled fetch path, an exhausted request deadline, an invalid or uncached URL, or a direct-fetch failure becomes a per-URL `failed_results` entry. Completed URLs remain in an HTTP 200 response. A non-deadline document-store failure remains an endpoint-level failure.
- One request resolves at most four of its maximum 20 URLs concurrently. Successful and failed arrays each preserve the relative order of their input URLs, and response-memory accounting is applied in input order rather than worker completion order.
- A fatal storage error or recovered worker panic cancels and joins all request-owned resolution workers before the endpoint returns. Panic logging uses a stable message and does not expose the recovered dependency value.
- Basic and advanced extract defaults remain 10 and 30 seconds. The explicit range remains 1 through 60 seconds, subject to the existing 30-second raw-content ceiling and 16 MiB retained/output budgets.

### Current-source release validation

- Direct `make e2e-node` now builds the node image before it runs, and `make e2e-crawler` builds both node and crawler images before it runs. Aggregate and parallel Make invocation can no longer satisfy the release gate from stale local product tags.

## Compatibility

- No YaCy request or response field, RWI form, crawler RPC field, stored document encoding, anchor publication encoding, Bleve mapping, Tavily envelope, route, setting key, setting default, setting value format, Admin control, environment variable, listener address, port, package path, service, image name, deployment topology, or third-party dependency changes.
- YaCy hello wire behavior is preserved, but accepted caller observations now affect the local roster. A locally failed callback is persisted as junior rather than discarded, and a later successful callback can promote it. Junior exclusion is enforced independently at active, seed-list, search, reachable, and DHT boundaries.
- Outbound DHT gate behavior changes intentionally. A derived loopback or local-listener query no longer claims public reachability. Current peer evidence is authoritative, including an all-junior negative state; a pinned direct query is used only when no peer evidence exists. A node with neither current eligible peer evidence nor `YAGO_PUBLIC_SELF_TEST_URL` remains unconfirmed and keeps public-reachability-dependent DHT distribution closed.
- The Admin Network and Overview pages expose the new evidence-backed states without changing their routes, forms, pagination, or no-JavaScript behavior.
- The `yago-crawler` runtime does change: obsolete stream attempts now cancel completely after node-reported session loss. There is no contract or checkpoint migration, but node and crawler binaries should be upgraded together under the existing release policy.
- `YAGO_WEB_FALLBACK_SEED_CRAWL` and `YAGO_EXTRACT_FETCH_ENABLED` remain disabled by default. Their names, validation, defaults, Admin controls, and related timeout and size settings are unchanged.
- The extract response remains Tavily-shaped: successes are in `results`, expected URL-specific failures are in `failed_results`, and fatal endpoint errors retain the detail-only error envelope. Concurrency does not reorder either array.
- Existing documents, anchors, peer rows, crawl orders, leases, worker checkpoints, and frontiers remain readable. Installation starts no migration, index rebuild, recrawl, or warming order solely because of the upgrade.

## Upgrade and rollback

1. After the immutable tag workflow succeeds, download the architecture-matching v0.0.20 package. On `root@yagoseek.dev`, stage it under `/opt/yago/releases/v0.0.20` and verify that staged file's GitHub artifact attestation against repository `D4rk4/yago`, workflow `.github/workflows/release.yml`, tag ref `refs/tags/v0.0.20`, and the tagged source commit before stopping either service. Other targets verify the package in their selected staging directory.
2. Follow the target's data-protection policy. Generic bare-metal nodes take one coordinated stopped backup. The documented `root@yagoseek.dev` exception preserves every existing archive under `/opt/yago/backups` and creates no new pre-upgrade backup.
3. Stop `yago-crawler` and then `yago-node`. Install the package with `dpkg -i` and confirm that package installation did not start either service automatically.
4. Start `yago-node` first. Verify its exact version, `/health`, `/ready`, all configured HTTP listeners, and Admin Network reachability provenance. An unconfirmed DHT gate is expected until an eligible peer observation arrives unless an explicit public self-test URL is configured.
5. Start `yago-crawler` after node readiness. Verify its exact version, control and ingest connections, a current heartbeat, durable queue progress, and absence of a repeating lease-loss sequence.
6. Perform only observation-based production checks: service state, readiness, listener availability, current logs, crawler connections and progress, and read-only API responses. Exercise warming, uncached extraction, failures, and restart scenarios in the local Docker gate instead.

For rollback, stop crawler and node, reinstall the v0.0.19 package, start and verify the node, and then start the crawler. No data restore is required solely for format compatibility. Durable warming orders and junior seed rows use existing encodings and remain readable. However, v0.0.19 does not enforce the new locally junior candidate exclusions and may treat a retained junior row as an ordinary inactive peer; block such a row before rollback when strict exclusion is required. Rollback also restores the derived self-query reachability decision, the incomplete crawler stream cancellation, hash-stripe URL contention, extract lookup without the 250-millisecond sub-deadline, continuous-mutation capacity retry, duplicate warming root, and oversized anchor subphase.

## Known limitations

- External reachability evidence is process-local and expires after 15 minutes. A restart returns the state to unconfirmed until another eligible outgoing greeting completes or an explicitly pinned direct query runs.
- Only an observer with a public IP literal in its primary advertised address contributes external evidence. A valid observer advertised only by hostname or an IPv6-only secondary field is conservatively ignored.
- Any current peer evidence suppresses the direct fallback. An all-junior set can therefore keep the DHT public-reachability gate closed until those observations expire or a later successful greeting replaces one of them. This is fail-closed behavior, not an attempt to reconcile conflicting probes.
- A pinned direct query still originates from the node itself. A NAT without hairpin support can make that fallback fail even when an external client can connect; peer back-ping evidence remains the stronger source.
- The potential-peer roster contains only callers the node has observed through authenticated inbound hello. It is not a complete census of junior peers on the YaCy network, and juniors deliberately contribute no search or routing capacity.
- The 4,096-peer reservoir is shared with ordinary peer observations. A stream
  of authenticated callers can churn older retained observations under the
  existing eviction policy, although locally junior entries still cannot become
  active, searchable, exportable, or routable without a successful callback.
- Listener prebinding removes the local startup race for HTTP surfaces; it cannot prevent a remote observer from probing during an unrelated reverse-proxy, firewall, or NAT transition.
- The 128 pending warming jobs are process memory, not durable frontier state. Queue saturation, shutdown, or process failure before successful `PublishOnce` delivery can lose optional warming work while preserving the search response.
- Web warming is asynchronous. A surfaced URL is not locally extractable until crawler fetch, parse, ingest, and index complete. Robots, egress rejection, page budgets, unsupported formats, remote failure, and content policy can prevent storage.
- A 50-millisecond presence timeout can add an idempotent dispatcher attempt for a document already being stored. It cannot create a second accepted order for the same URL, but it can consume lookup and duplicate-detection work.
- The URL registry bounds idle entries, not simultaneous identities. More than 4,096 active or referenced URLs can temporarily keep more boundaries. Main-vault, filesystem, Bleve, and crawler gates remain shared sources of bounded backpressure.
- A capacity observation reused for up to one second can be slightly older than a following write. The soft quota and storage-pressure gates remain advisory admission controls rather than an atomic filesystem quota.
- Anchor replacement is page-recoverable, not cross-store atomic. A failure after an earlier target page commits can expose an intermediate projection before publication; a failure in a later final-publication subtransaction can leave earlier source rows published and later rows unchanged. At-least-once redelivery recomputes the remaining aggregate and converges both states, but can repeat reads and checks from completed work.
- The 250-millisecond lookup boundary reserves time but cannot guarantee that DNS, connection, response, parsing, and extraction finish inside the remaining request budget. Fetch-on-extract remains disabled by default, non-deadline storage failures remain endpoint errors, and multi-URL extraction remains bounded but non-streaming.
- Duplicate URLs in one extract request are not coalesced. Each of the at most 20
  input positions resolves independently under the four-worker and response
  budgets, and repeated inputs remain repeated in input-relative output order.
- Independent restart recovery preserves at-least-once delivery. Work whose outcome was not committed can run again, and recovery still depends on the node, crawler policy, durable lease state, and transport becoming available within their existing retry behavior.
